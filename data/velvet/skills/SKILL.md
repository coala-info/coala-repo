# velvet

## Overview

Velvet is a suite of algorithms designed to manipulate de Bruijn graphs for genomic assembly. It is optimized for short-read sequencing data and consists of two core programs: **velveth** and **velvetg**. The process is sequential: `velveth` constructs the dataset by hashing reads into k-mers, and `velvetg` builds the graph, simplifies it, and generates the final contigs.

## Core Workflow

### 1. Data Preparation (velveth)
The `velveth` command creates a roadmap for the assembly by hashing the reads.

**Basic Syntax:**
`velveth <output_directory> <kmer_size> [-file_format] [-read_type] <filename>`

*   **kmer_size**: Must be an odd integer. Smaller k-mers increase sensitivity but also increase graph complexity and memory usage.
*   **File Formats**: `-fasta`, `-fastq`, `-fasta.gz`, `-fastq.gz`, `-sam`, `-bam`, `-raw`.
*   **Read Types**: `-short`, `-shortPaired`, `-short2`, `-shortPaired2`, `-long`, `-longPaired`.

**Example:**
`velveth assembly_dir 31 -fastq -shortPaired reads_1.fq reads_2.fq`

### 2. Assembly Generation (velvetg)
The `velvetg` command performs the actual assembly using the files generated by `velveth`.

**Basic Syntax:**
`velvetg <output_directory> [options]`

**Common Options:**
*   `-exp_cov <auto|int>`: Expected coverage of the genome. Setting this helps Velvet distinguish between repeats and unique regions.
*   `-cov_cutoff <auto|float>`: Removes low-coverage nodes (potential sequencing errors).
*   `-ins_length <int>`: Expected insert length for paired-end reads.
*   `-min_contig_lgth <int>`: Minimum length of contigs exported to `contigs.fa` (default is 2*k).

**Example:**
`velvetg assembly_dir -exp_cov auto -cov_cutoff 5 -ins_length 300`

## Expert Tips and Best Practices

### Memory Management
Velvet is memory-intensive. For large genomes, ensure you are running on a 64-bit Linux environment.
*   **Requirement**: At least 12GB of RAM is recommended for initial small-to-medium assemblies.
*   **Optimization**: If memory is a bottleneck, consider using a larger k-mer size to reduce the number of nodes in the de Bruijn graph.

### Parallelization
Velvet supports OpenMP for multi-threading.
*   Set the environment variable `OMP_NUM_THREADS` to the number of available CPU cores before running `velveth` or `velvetg`.
*   **Warning**: Some versions may stall if `OMP_THREAD_LIMIT` is set to 1.

### K-mer Selection
*   The default maximum k-mer length is often 31. If you need a k-mer length > 31, Velvet must be recompiled with `CATEGORIES=N` and `MAXKMERLENGTH=M` flags in the Makefile.
*   Always use odd numbers for k-mer sizes to avoid palindromic k-mers that can confuse the graph construction.

### Colorspace Assembly
For SOLiD data, Velvet must be compiled specifically for colorspace:
`make color`

## Troubleshooting Common Issues

*   **Segmentation Faults**: Often caused by insufficient physical memory or attempting to process paired-end data with mismatched read counts.
*   **Word Length Errors**: If you receive "Word length X greater than max allowed value", you must recompile the source code with a higher `MAXKMERLENGTH`.
*   **Empty Contigs**: Usually indicates the `cov_cutoff` is set too high or the `kmer_size` is too large for the sequencing depth provided.
