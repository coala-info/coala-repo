# chipseq-greylist

## Overview

The `chipseq-greylist` tool is a Python implementation of the Bioconductor `GreyListChIP` algorithm. While standard ENCODE blacklists provide a set of known problematic regions across the genome, they do not account for artifacts unique to a specific library or sample. This tool analyzes an input (control) BAM file to statistically determine a coverage threshold and output a BED file of regions exceeding that threshold. By filtering peaks that overlap with these "greylist" regions, you can significantly reduce the false discovery rate in your ChIP-seq analysis.

## Command Line Usage

The tool is designed for simplicity and typically requires only the input BAM file as an argument.

### Basic Execution
Run the tool on your control/input BAM file:
```bash
chipseq-greylist input_sample.bam
```

### Output Files
The command generates three primary files using the BAM filename as a prefix:
1.  `input_sample-input-grey.bed`: The final greylist regions in BED format.
2.  `input_sample-input-greystats.csv`: Contains bootstrapped negative binomial parameters and the calculated coverage threshold.
3.  `input_sample-input-greydepth.tsv`: The windowed depth data used for the calculation.

## Downstream Filtering

Once the greylist BED file is generated, use `bedtools` to annotate or filter your peak calls.

### Annotating Peaks
To see which peaks overlap with greylist regions without removing them:
```bash
bedtools intersect -wao -a sample_peaks.bed -b input_sample-input-grey.bed > peaks_annotated.bed
```

### Filtering Peaks
To create a "clean" peak set by removing any peak that overlaps with a greylist region:
```bash
bedtools intersect -v -a sample_peaks.bed -b input_sample-input-grey.bed > peaks_filtered.bed
```

## Best Practices and Expert Tips

*   **Input Only**: Always run this tool on the **Input/Control** BAM file, not the ChIP sample. The goal is to identify regions where the background model fails due to excessive depth.
*   **Complementary to Blacklists**: Greylists are sample-specific. You should still use standard ENCODE blacklists (e.g., for hg38 or mm10) in addition to the greylist generated by this tool.
*   **Library Complexity**: If `chipseq-greylist` flags an unusually large portion of the genome, it may indicate a library preparation issue or extremely low library complexity in your input sample.
*   **Peak Caller Integration**: Most peak callers (like MACS3) attempt to model background noise using the input sample, but they can still be "tricked" by extremely high-depth regions. Pre-filtering or post-filtering with a greylist provides a more robust statistical safeguard.
