# bioconductor-gqtlbase

## Overview

The `gQTLBase` package provides the fundamental data structures and interrogation methods for large-scale genomic QTL searches. It is designed to handle the billions of test results generated by genome-wide cis-association studies (like GEUVADIS) by using a "sharded" archive approach. The central object is the `ciseStore`, which wraps a `BatchJobs` registry to allow for efficient parallel processing and targeted retrieval of SNP-gene association statistics without loading the entire dataset into memory.

## Core Workflows

### 1. Accessing a QTL Store
To interact with QTL results, you typically start with a `ciseStore` object.

```r
library(gQTLBase)
# Example using geuvStore2 data
library(geuvStore2)
mm = makeGeuvStore2()
print(mm) # Displays registry status and an excerpt of results
```

### 2. Targeted Data Extraction
You can extract specific association results by probe (gene) ID or by genomic range.

**By Probe ID:**
```r
# Extract results for specific Ensembl IDs
pvec = mm@probemap[1:4, 1] 
litex = extractByProbes(mm, pvec)
# Returns a GRanges object with chisq and permutation scores
```

**By Genomic Range:**
```r
# Extract results within a specific genomic region
my_range = GRanges("1", IRanges(500000, 600000))
range_results = extractByRanges(mm, my_range)
```

### 3. Parallel Processing with storeApply
To perform operations across the entire archive (e.g., calculating lengths, filtering, or custom transformations), use `storeApply`. This integrates with `BiocParallel`.

```r
library(BiocParallel)
register(MulticoreParam(workers = 4))

# Apply a function (e.g., length) to every job in the store
lens = storeApply(mm, length)
summary(unlist(lens))
```

### 4. Memory-Efficient Statistics with ff
For operations requiring access to all association statistics (like FDR calculation or quantile estimation), use `storeToFf` to create a disk-backed `ff` object.

```r
# Extract all chi-squared statistics into an ff vector
allassoc = storeToFf(mm, "chisq")
length(allassoc)
# Accessing elements does not load the whole vector into RAM
print(allassoc[1:10])
```

## Key Functions Reference

- `ciseStore`: The primary class for managing sharded QTL results.
- `extractByProbes(store, probeids)`: Retrieves all association tests for specific identifiers.
- `extractByRanges(store, ranges)`: Retrieves tests falling within specified genomic coordinates.
- `storeApply(store, FUN, ...)`: Applies a function to results of each job in the store in parallel.
- `storeToFf(store, field)`: Maps a specific metadata column (e.g., "chisq") from all results into an `ff` file for high-performance analysis.

## Tips for Success
- **BatchJobs Integration**: Since `ciseStore` relies on `BatchJobs`, you can use `loadResult(mm@reg, id)` to inspect individual job outputs directly if needed.
- **Downstream Analysis**: Use the `gQTLstats` package for advanced FDR estimation and visualization based on the `ciseStore` objects created or managed here.
- **Memory Management**: Always prefer `extractByProbes` or `storeToFf` over trying to load the entire registry into a standard R list, as these datasets often exceed available RAM.
