# perl-par

## Overview

The `perl-par` skill focuses on the use of the `PAR::Packer` suite, primarily the `pp` (Perl Packer) utility. This tool is essential for Perl developers who need to distribute their software to environments where installing CPAN modules is difficult or where a specific Perl version is required. It works by scanning your script for dependencies and bundling them into a single, executable file or a `.par` archive. This ensures that the application runs with the exact versions of the libraries it was developed with, significantly reducing "works on my machine" deployment issues.

## Common CLI Patterns

### Creating Stand-alone Executables
The most common use case is generating a binary that runs without requiring a pre-installed Perl interpreter or modules on the target system.
```bash
pp -o my_app.exe script.pl
```

### Bundling Additional Resources
If your script relies on external files (config, images, templates) that aren't detected by static analysis, use the `-a` (addfile) flag.
```bash
pp -o my_app -a "config.yaml;config.yaml" -a "templates/;templates/" script.pl
```

### Forcing Module Inclusion
Static analysis may miss modules loaded via `eval` or dynamic strings. Use `-M` to force their inclusion.
```bash
pp -o my_app -M Module::Name -M Another::Module script.pl
```

### Creating PAR Archives
If the target machine has Perl and `PAR` installed, you can create a smaller `.par` file instead of a full binary.
```bash
pp -p -o my_app.par script.pl
```

### Self-Contained Perl Scripts
To create a script that is still plain text but contains all dependencies in a compressed format at the end:
```bash
pp -P -o packed_script.pl source.pl
```

## Expert Tips and Best Practices

### Handling Shared Libraries
If your Perl was built with a shared library (e.g., `libperl.so` or `perl5x.dll`), the stand-alone executable may still require that library to be present on the target system. Always test your binary on a "clean" virtual machine or container that mimics the production environment.

### Troubleshooting Missing Dependencies
If the packaged application fails with "Can't locate ... in @INC", use the `-v` (verbose) flag during packing to see which modules `pp` is detecting.
```bash
pp -v 3 -o my_app script.pl
```

### Temporary Directory Management
PAR executables extract their contents to a temporary directory (usually `cache-user` in the system temp folder) before execution. 
- You can influence this location by setting the `PAR_TMPDIR` environment variable.
- Use the `--clean` flag if you want the temporary files to be removed immediately after the script finishes, though this increases startup time for subsequent runs.

### Platform Specificity
`PAR::Packer` does **not** support cross-compilation. To create a Windows `.exe`, you must run `pp` on Windows. To create a Linux binary, you must run it on Linux.

### macOS Code Signing
On modern macOS versions, stand-alone executables generated by `pp` may require manual code signing to run without security warnings or immediate termination by the OS.
