# mags-taxonomy-annotation

## Overview

This workflow performs taxonomic classification of Metagenome-Assembled Genomes (MAGs) using [GTDB-Tk](https://github.com/Ecogenomics/GTDBTk). It is designed to process sequence collections—such as bins generated by SemiBin2, MetaBAT2, or dRep—and reconcile differences between classification systems by mapping GTDB lineages to NCBI nomenclature.

The process begins by running the GTDB-Tk classify workflow to generate taxonomic summaries. A series of text-processing steps, including [awk](https://www.gnu.org/software/gawk/) and [sed](https://www.gnu.org/software/sed/) reformatting, extracts the GTDB lineages and ranks. These are then passed through mapping tools to translate GTDB names into corresponding NCBI names and resolve them into official NCBI taxIDs.

The final output is a comprehensive merged table containing BinIDs, GTDB names, taxonomic ranks, NCBI names, and NCBI taxIDs. Additionally, the workflow generates [MultiQC](https://multiqc.info/) reports and intermediate mapping files to ensure full traceability of the taxonomic assignments. This workflow is released under the [MIT license](https://opensource.org/licenses/MIT).

## Inputs and data preparation

| Step | Label | Type | Notes |
| --- | --- | --- | --- |
| 0 | Sequence collection | data_collection_input | Provide a collection of MAGs from dREP or another binning tool. |
| 1 | GTDB-tk Database | parameter_input | GTDB-tk Database used for Bin Taxonomy  Classification |


Ensure your input MAGs are organized into a Galaxy dataset collection (list) containing FASTA files, such as those generated by MetaBAT2 or SemiBin2. You must also provide the appropriate GTDB-tk reference database as a parameter input to ensure successful classification. For automated execution, you can initialize your configuration using `planemo workflow_job_init` to generate a `job.yml` file. Refer to the README.md for comprehensive details on the specific column structures and the mapping logic between GTDB and NCBI taxonomies. All intermediate text processing is handled automatically to produce the final merged mapping table.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run MAGs-taxonomy-annotation.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run MAGs-taxonomy-annotation.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run MAGs-taxonomy-annotation.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init MAGs-taxonomy-annotation.ga -o job.yml`
- Lint: `planemo workflow_lint MAGs-taxonomy-annotation.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
