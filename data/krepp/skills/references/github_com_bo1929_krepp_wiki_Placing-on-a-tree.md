#### Phylogenetic placement on a backbone tree

Since the references in a given index also share an evolutionary history, finding a placement on a phylogeny (or, less ideally, on a taxonomy) might provide better downstream performance and interpretability.
`krepp place` subcommand performs phylogenetic placement on a backbone tree, which is either used to build the target index or specified during the query time (both using the `-t,--nwk-file` option) as follows:

```
krepp place -i /path/to/index -q /path/to/query.fastq -t /path/to/backbone.nwk --num-threads 16 -o /path/to/output.jplace
```

The output, by default, is an information-rich JSON, namely the jplace format (version 3) (see the description [here](https://doi.org/10.1371/journal.pone.0031009)). Each placed read is a list entry under the `placements` field. For each placed read, `n` field stands for the read ID, and `p` is another list of potentially multiple placements represented by several values:

1. `edge_num`: the placement edge, the jplace file contains a Newick tree decorated with edge numbers in curly brackets
2. `pendant_length`: the branch length for the placement edge, can be negative.
3. `distal_length`: the length from the distal (away from the root) edge to the placement location (mid-point by default)
4. `likelihood`: the likelihood of the distance estimation (not comparable across placements)
5. `like_weight_ratio`: weight of placements if there are multiple equally good placements for a sequence
6. `distance`: estimated distance to the clade under the placement edge

krepp does not report unplaced sequences, but the total number of queries can be found in the `metadata` field of the jplace file. We recommend [gappa](https://github.com/lczech/gappa) for analyzing and visualizing jplace files. Several other tools, such as [tax2tree](https://github.com/biocore/tax2tree) and [treeio](https://rdrr.io/bioc/treeio/), can also process jplace files.

Alternatively, you can use `--tabular` option to report placements in a tab-separated format, where each row corresponds to a placement, and hence, a read can appear in multiple rows. The format is quite intuitive:

#### Configuring the placement algorithm

The default algorithm reports all placements with distances indistinguishable from the distance of the closest tip (`--multi`). If `--no-multi` is given, krepp reports only the *best* placement, which corresponds to the edge above the largest clade among potential placements.

The placement algorithm applies a more aggressive filtering compared to `krepp dist`. This is controlled by the `--filter` option, which is on by default. It leaves query sequences without `sufficient evidence` unplaced. krepp considers a placement weak if the number of *k*-mer matches below the Hamming distance $\tau$ (the option `--tau`, default: 2) is $\leq 1$. One can give the `--no-filter` flag to disable the filtering, or relax it by increasing the `--tau`.

#### Placing on a taxonomy

krepp is not primarily designed for taxonomic classification, but since its placement algorithm is branch length agnostic, it can place queries on a taxonomic tree, which is essentially a multifurcating tree without branch lengths. To do so, use the `-l` (`--lineage-file`) option instead of the `-t` (`--nwk-file`). This option expects taxonomic lineages in Greengenes2/GTDB format (see an example [here](https://github.com/bo1929/krepp/blob/master/test/lineages_toy.txt)). The lineage file is supposed to be a mapping between reference IDs as they appear in the index and their taxonomic labels at fixed ranks. By default, krepp converts these lineages into a multifurcating Newick tree and outputs a jplace file (alternatively, use `--tabular`).

More ideally, if you have a phylogeny and you are still interested in the taxonomic composition of your sample, you can decorate a Newick tree using [tax2tree](https://github.com/biocore/tax2tree). Then, with the `--tabular` flag, you can obtain placements on taxonomic lineages which would be in the `DISTAL_NODE` column.

#### Summarizing placements

If you would like to summarize all the placements to derive sequence abundances across the nodes of the given backbone tree, use the `--summarize` option.
This outputs both the weighted raw counts and normalized abundances.
Each placed read gets a count value of 1, shared across all placed edges, and unplaced reads are ignored in the normalization.
Note that these abundances cannot be interpreted as taxonomic abundance profiles as they are not aggregated. In other words, placements on an edge (or a taxonomic group) are not counted toward its parents.
