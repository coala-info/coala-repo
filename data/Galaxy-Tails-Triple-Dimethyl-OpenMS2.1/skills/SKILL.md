# tails-triple-dimethyl-openms21

## Overview

This workflow is designed for degradomics research, specifically for the detection and quantitation of N-termini using the N-TAILS (Terminal Amine Isotopic Labeling of Substrates) technique. It leverages the [OpenMS](https://www.openms.de/) framework within Galaxy to process triple dimethyl-labeled proteomic samples, enabling the identification of neo-N-termini generated by proteolytic activity.

The pipeline begins by processing mzML input collections and a UniProt FASTA database. It performs feature detection via `FeatureFinderMultiplex` and executes parallel peptide identifications using multiple `MSGFPlusAdapter` steps. High-resolution precursor mass correction and peptide indexing are applied to ensure high-accuracy matching against the reference database.

To ensure data quality, the workflow applies False Discovery Rate (FDR) estimation and stringent ID filtering. Identifications are merged and mapped back to detected features, with an automated conflict resolution step to handle overlapping assignments. The final results are consolidated and exported in the `MzTab` format, facilitating downstream statistical analysis of N-terminal peptides.

Ensure your mass spectrometry data is uploaded as an mzML dataset collection to maintain sample grouping, while the UniProt reference must be in FASTA format. Verify that your mzML files contain centroided data to ensure compatibility with the OpenMS FeatureFinderMultiplex and MSGFPlusAdapter tools. For automated execution, you can initialize your configuration using `planemo workflow_job_init` to generate a `job.yml` file. Refer to the README.md for comprehensive details on parameter settings and specific N-TAILS labeling requirements. Always check that your FASTA headers are compatible with the PeptideIndexer requirements to avoid identification errors.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run wf-ntails.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run wf-ntails.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run wf-ntails.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init wf-ntails.ga -o job.yml`
- Lint: `planemo workflow_lint wf-ntails.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
