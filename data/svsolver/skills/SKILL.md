# svsolver

## Overview
The `svsolver` suite is the computational engine of the SimVascular pipeline, designed for high-fidelity finite element Navier-Stokes simulations. It transforms anatomical meshes and physiologic boundary conditions into detailed hemodynamic data. This skill assists in navigating the command-line interface for the three core executables to ensure proper data flow from initial mesh preparation to final result extraction.

## Core Executables
The suite consists of three primary programs that must be run in sequence:

1.  **svpre (Presolver)**: Processes the mesh and boundary condition files into a format the solver can ingest.
2.  **svsolver (Flowsolver)**: The parallelized Navier-Stokes solver that computes fluid velocities and pressures.
3.  **svpost (Postsolver)**: Converts raw solver output into visualization formats (e.g., VTK) for analysis.

## Common CLI Patterns

### 1. Preprocessing with svpre
Before running a simulation, use `svpre` to prepare the calculation files.
- **Input**: Requires a `.svpre` script file containing paths to mesh files, fluid properties, and boundary conditions.
- **Command**: `svpre <filename>.svpre`

### 2. Running the Flowsolver
The solver is typically executed using MPI for parallel performance.
- **Input**: Requires the `solver.inp` file (generated by svpre or manually configured) and the `geombc.dat.1` file.
- **Standard Execution**: `svsolver solver.inp`
- **Parallel Execution**: `mpirun -np <num_procs> svsolver solver.inp`

### 3. Post-processing with svpost
After the simulation completes, use `svpost` to aggregate results.
- **Common Task**: Converting restart files to VTK for ParaView.
- **Command**: `svpost -all -vtu -start <step> -stop <step> -incr <step>`
- **Key Flags**:
    - `-all`: Process all available data fields.
    - `-vtu`: Output in XML VTK format.
    - `-vtp`: Output surface data.

## Expert Tips
- **File Naming**: Ensure that `geombc.dat.1` and `restart.0.1` are in the working directory before starting `svsolver`; the solver expects specific naming conventions for initial conditions.
- **Convergence Monitoring**: Check the `history.dat` file during execution to monitor residual convergence.
- **Memory Management**: For large meshes (>1M elements), ensure your MPI configuration allocates sufficient memory per core, as the finite element assembly is memory-intensive.
