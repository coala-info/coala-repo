# brew3r

## Overview

BREW3R (**B**ulk **R**NA-seq **E**vidence-based **W**orkflow for **3**' UTR **R**eannotation) is designed to extend existing gene annotations (GTF) using *de novo* assembly from full-length bulk RNA-seq data. It is particularly useful for refining 3' UTR boundaries based on empirical evidence from BAM files, such as those generated by STAR.

The workflow utilizes [StringTie](https://toolshed.g2.bx.psu.edu/repos/iuc/stringtie/stringtie/2.2.3+galaxy0) to perform initial transcript assembly for each input BAM file, followed by a merging step to create a unified *de novo* assembly. The core logic is driven by the [BREW3R.r](https://bioconductor.org/packages/release/bioc/html/BREW3R.r.html) R package, which integrates the merged transcripts with the reference GTF to produce an extended annotation file.

Users must provide a collection of BAM files, a reference GTF, and specify parameters including library strandedness, minimum coverage for assembly, and minimum FPKM for the merging process. For unstranded libraries, the workflow automatically filters out ambiguous transcripts that overlap exons on both strands to ensure annotation accuracy.

Ensure your input GTF and BAM files are correctly formatted, as the workflow relies on a paired collection of BAM files to perform *de novo* assembly via StringTie. You must specify the library strandedness accurately—choosing between forward, reverse, or unstranded—to ensure BREW3R.r correctly assigns transcript orientations during the extension process. Adjust the minimum coverage and FPKM thresholds to filter the sensitivity of the merged assembly based on your specific sequencing depth. For automated execution, you can initialize your environment using `planemo workflow_job_init` to generate a `job.yml` template. Refer to the README.md for comprehensive details on parameter defaults and specific file preparation requirements.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run BREW3R.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run BREW3R.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run BREW3R.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init BREW3R.ga -o job.yml`
- Lint: `planemo workflow_lint BREW3R.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
