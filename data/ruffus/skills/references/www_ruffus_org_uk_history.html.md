However, when we need to place the output in a different directories, we suddenly have to plunge into the deep end and parse file paths using regex() or formatter() . Now, &#64;transform takes an optional output_dir named parameter so that we can continue to use suffix() even when the output needs to go into a new directory. # # input/a.fasta -&gt; output/a.sam # input/b.fasta -&gt; output/b.sam # starting_files = [ &quot;input/a.fasta&quot; , &quot;input/b.fasta&quot; ] @transform ( starting_files , suffix ( &#39;.fasta&#39; ), &#39;.sam&#39; , output_dir = &quot;output&quot; ) def map_dna_sequence ( input_file , output_file ) : pass See example test\test_suffix_output_dir.py 2) Named parameters ¶ Decorators can take named parameters. These are self documenting, and improve clarity. Note that the usual Python rules for function parameters apply: Positional arguments must precede named arguments Named arguments cannot be used to fill in for &#8220;missing&#8221; positional arguments For example the following two functions are identical: Positional parameters: @merge ( prev_task , [ &quot;a.summary&quot; , &quot;b.summary&quot; ], 14 , &quot;extra_info&quot; , { &quot;a&quot; : 45 , &quot;b&quot; : 5 }) def merge_task ( inputs , outputs , extra_num , extra_str , extra_dict ): pass Named parameters: # new style is a bit clearer @merge ( input = prev_task , output = [ &quot;a.summary&quot; , &quot;b.summary&quot; ], extras = [ 14 , &quot;extra_info&quot; , { &quot;a&quot; : 45 , &quot;b&quot; : 5 }] ) def merge_task ( inputs , outputs , extra_num , extra_str , extra_dict ): pass Warning ,extras= takes all the extras parameters ( 14, &quot;extra_info&quot;, {&quot;a&quot;:45, &quot;b&quot;:5} ) as a single list &#64;split(...) and &#64;merge(...) input output [ extras ] &#64;transform(...) and &#64;mkdir(...) input filter [ replace_inputs or add_inputs ] output [ extras ] [ output_dir ] &#64;collate(...) and &#64;subdivide(...) input filter output [ extras ] &#64;originate(...) output [ extras ] &#64;product(...) , &#64;permutations(...) , &#64;combinations(...) , and &#64;combinations_with_replacement(...) input filter [ input2...NNN ] (only for product ) [ filter2...NNN ] (only for product ) where NNN is an incrementing number tuple_size (except for product ) [ replace_inputs or add_inputs ] output [ extras ] 3) New object orientated syntax for Ruffus ¶ Ruffus Pipelines can now be created directly using the new Pipeline and Task objects instead of via decorators. # make ruffus pipeline my_pipeline = Pipeline ( name = &quot;test&quot; ) my_pipeline . transform ( task_func = map_dna_sequence , input = starting_files , filter = suffix ( &#39;.fasta&#39; ), output = &#39;.sam&#39; , output_dir = &quot;output&quot; ) my_pipeline . run () This new syntax is fully compatible and inter-operates with traditional Ruffus syntax using decorators. Apart from cosmetic changes, the new syntax allows different instances of modular Ruffus sub-pipelines to be defined separately, in different python modules and then joined together flexible at runtime.
