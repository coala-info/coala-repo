# Proteomics: MaxQuant and MSstats for the analysis of label-free data

## Overview

This Galaxy workflow provides a comprehensive pipeline for the downstream analysis of label-free proteomics data, specifically designed to process outputs from MaxQuant. It automates the transition from raw protein identification files to statistical significance testing and biological visualization. The workflow is based on [GTN training materials](https://training.galaxyproject.org/) and is optimized for identifying differentially expressed proteins in comparative studies.

The process begins by ingesting MaxQuant `proteinGroups` and `evidence` files, alongside user-provided annotation and comparison matrices. It utilizes several text-processing steps to clean and format the data before passing it to [MSstats](https://msstats.org/), which performs statistical modeling and group comparisons. The workflow then filters the results to isolate significant proteins based on fold-change and p-value thresholds.

In the final stages, the pipeline integrates with the [UniProt](https://www.uniprot.org/) REST interface to retrieve functional annotations for the identified proteins. It generates high-quality visualizations, including heatmaps via `ggplot2`, to highlight upregulated proteins across different experimental conditions. This end-to-end approach ensures that researchers can move efficiently from quantitative tables to biological insights.

Ensure your input files are in tabular format, specifically using the `proteinGroups.txt` and `evidence.txt` files generated by MaxQuant alongside a custom annotation file and comparison matrix. Verify that column headers match the expected MSstats format to avoid errors during the text replacement and filtering steps. For large-scale studies, organizing these inputs into a Galaxy dataset collection can streamline processing, though individual datasets are sufficient for this specific workflow. Refer to the `README.md` for comprehensive details on constructing the comparison matrix and annotation file. You can use `planemo workflow_job_init` to generate a `job.yml` for automated execution.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run main-workflow.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run main-workflow.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run main-workflow.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init main-workflow.ga -o job.yml`
- Lint: `planemo workflow_lint main-workflow.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
