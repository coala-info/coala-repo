# piper

## Overview
Piper (pypiper) is a specialized Python toolkit designed to "glue" together command-line tools into robust, restartable pipelines. Its core strength lies in its ability to track the progress of a workflow; if a pipeline is interrupted or fails, it can be resumed from the last successful stage, skipping steps that have already been completed. This makes it ideal for long-running computational tasks, such as bioinformatics processing, where re-running early stages is time-prohibitive.

## Installation
Install the package via Conda from the bioconda channel:
```bash
conda install bioconda::piper
```

## Core CLI Patterns
When running a pipeline built with piper, use these standard flags to control execution flow:

- **Restart from scratch**: Use `-N` or `--new-start` to ignore existing checkpoints and force the pipeline to run from the very first stage.
- **Overwrite outputs**: Use `--clobber` or `--overwrite` when you need to re-run stages and replace existing files.
- **Logging**: Piper integrates with `logmuse`. Ensure your environment is configured to capture logs, as piper tracks "failure reasons" which are critical for debugging interrupted runs.

## Expert Tips and Best Practices

### Path Management
- **Use Absolute Paths**: Always set the pipeline output folder to an absolute path. Using relative paths can cause issues with the checkpointing mechanism, preventing the pipeline from correctly identifying completed stages upon restart.
- **Avoid Log Collisions**: Ensure the pipeline log file path does not collide with filepaths provided to other logging utilities (like logmuse) to prevent write errors or truncated logs.

### Workflow Robustness
- **Checkpointing**: Piper automatically checkpoints stages. If a stage is marked as `nofail`, it may not trigger a checkpoint if it fails but is allowed to continue. Be mindful of downstream stages that depend on the output of a `nofail` stage.
- **Stage Granularity**: Design pipelines with discrete `Stage` objects. This allows for more granular restarts and better reporting of where a failure occurred.
- **Resource Monitoring**: For huge files (e.g., large FASTQ files in bioinformatics), piper is designed to handle the shell-level execution, but ensure the underlying system has sufficient disk space for the checkpoint files generated by the toolkit.

### Integration with Pipestat
Piper can report results directly to `pipestat`. When this is enabled:
- Use a unique `record_identifier` (often a sample name) to track results.
- The `pipestat_results_file` will store the status and results of the pipeline run, providing a structured way to query pipeline success beyond simple log files.
