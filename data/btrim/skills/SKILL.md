# btrim

## Overview

btrim is a specialized tool designed to refine compacted de Bruijn graphs by trimming "tips"—short, dead-end paths that usually represent sequencing errors rather than true biological variation. It is a core component of the Bcool short-read correction pipeline but can be used independently to process any compatible unitig graph. By removing these artifacts, btrim reduces graph complexity and improves the reliability of downstream assembly or mapping tasks.

## Command Line Usage

The basic syntax for btrim follows a positional argument structure:

```bash
btrim <unitig_file> <kmer_size> [tipping_length] [cores] [hash_size]
```

### Arguments

- **unitig_file**: The input graph file, typically in a format produced by BCALM2.
- **kmer_size**: The k-mer length used to construct the initial de Bruijn graph.
- **tipping_length**: (Default: 100) The maximum length of a dead-end path to be removed. Paths longer than this are preserved as potential biological features.
- **cores**: (Default: 1) The number of threads to use for processing.
- **hash_size**: (Default: 8) Controls the number of intermediate files created (2^h). Increasing this can help with memory management on very large datasets.

## Best Practices and Expert Tips

### Input Preparation
btrim is optimized to work with unitig files generated by **BCALM2**. Ensure your graph is compacted before running btrim to ensure the "tips" are correctly identified as terminal unitigs.

### Tuning Tipping Length
The default tipping length of 100 is generally effective for standard Illumina short-read data. However:
- **For high-error data**: Consider increasing the tipping length if the graph remains overly complex.
- **For short k-mers**: If using very small k-mers, a smaller tipping length may be necessary to avoid over-trimming real sequences.
- **Rule of thumb**: The tipping length should typically be at least 2x the k-mer size but less than the average read length.

### Performance Optimization
- **Multithreading**: While btrim supports multiple cores, the efficiency of parallelization depends on the `hash_size`. If you increase the core count, ensure the `hash_size` is sufficient to distribute the workload effectively.
- **Memory Management**: If you encounter memory exhaustion on large graphs, increase the `hash_size` parameter (e.g., to 10 or 12) to use more temporary files and reduce the per-thread memory footprint.

### Advanced Filtering
Based on the tool's development history, btrim supports specific modes for different biological contexts:
- **RNA-Seq/Transcriptomics**: Use the tool's "trans" or "RNA" modes if available in your version to handle the uneven coverage profiles typical of transcriptomic data, which prevents the accidental removal of low-abundance isoforms.
- **Abundance Filtering**: You can apply abundance filters (min/max) to remove unitigs that fall outside expected coverage ranges, further cleaning the graph of systematic errors or highly repetitive sequences.
