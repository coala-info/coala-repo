## Typical command lines

Short-read paired-end mapping from an **interleaved FASTQ file** to GAM format using 32 threads:

```
vg giraffe -p -t 32 -Z graph.gbz -d graph.dist -z graph.shortread.zipcodes -m graph.shortread.withzip.min \
    -i -f reads.fq.gz -b default > output.gam
```

Short-read paired-end mapping from **two FASTQ files** to GAM format using 32 threads:

```
vg giraffe -p -t 32 -Z graph.gbz -d graph.dist -z graph.shortread.zipcodes -m graph.shortread.withzip.min \
    -f reads.R1.fq.gz -f reads.R2.fq.gz -b default > output.gam
```

Short-read paired-end mapping from an interleaved FASTQ file to **BAM format** using 32 threads:

```
vg giraffe -p -t 32 -Z graph.gbz -d graph.dist -z graph.shortread.zipcodes -m graph.shortread.withzip.min -x graph.xg \
    -i -f reads.fq.gz -o BAM -b default > output.bam
```

**Long read** (`-b hifi` or `-b r10`) single-end mapping from a FASTQ file to GAM format using 32 threads:

```
vg giraffe -p -t 32 -Z graph.gbz -d graph.dist -z graph.longread.zipcodes -m graph.longread.withzip.min \
    -f reads.fq.gz -b hifi > output.bam
```

## Inputs

### Graphs

Giraffe uses a GBZ graph that combines the graph with haplotype information. The graph must be provided with option `-Z` / `--gbz-name`. Other graph types can be converted to GBZ format; e.g. use `vg gbwt -G` to produce a GBZ from a GFA.

If you start from a graph in GFA format, use W-lines for the haplotype paths. The heuristics vg uses with P-lines are fragile and can lead to incorrect results. In particular, in version 1.52.0 and earlier, `vg autoindex` does not understand P-lines properly. Starting from **version 1.53.0**, `vg autoindex` will recognize haplotypes stored as P-lines, as long as path names follow one of the [expected patterns](https://github.com/vgteam/vg/wiki/Path-Metadata-Model).

**Graph construction is a hard problem.** A pangenome graph is effectively a multiple sequence alignment. While it is easy to build an alignment, finding a useful alignment may require trial and error. Giraffe prefers graphs that avoid complex structures at every level of detail, while simultaneously minimizing sequence duplication. Complex structures include high-degree nodes, collapsed repetitive regions, and long-distance edges.

The [Minigraph–Cactus pipeline](https://github.com/ComparativeGenomicsToolkit/cactus/blob/master/doc/pangenome.md) builds graphs like that, and the default parameters are suitable for human-like genomes. With a small number of haplotypes (e.g. 10), the default (`clip`) graph is usually a good choice. With substantially more haplotypes, you may get better results with the `filter` graph that drops nodes used only by a single haplotype.

#### Handling long nodes

VG does not work well with graphs that contain long nodes. In particular, a **minimizer index cannot be built if there are nodes longer than 1024 bp**. If you build/index a graph with `vg autoindex`, build a graph with `vg construct`, or convert a GFA graph to GBZ with `vg gbwt -G`, long nodes are automatically chopped into sufficiently short pieces.

If the nodes of a GFA graph were chopped (or their ids converted from strings to integers), the resulting GBZ graph contains a translation from the chopped graph to the original graph. With option `--named-coordinates`, Giraffe will output alignments in the original graph. Some other vg commands also have similar options, but you should avoid using them. Having two incompatible versions of the graph is confusing and error-prone, and the translation is not guaranteed to survive operations that change the graph.

Instead, **you should stick to the vg version of the graph**. If you need a compatible GFA file for other tools, you can generate it with:

```
vg convert -f --no-translation graph.gbz > graph.gfa
```

### Distance index, zipcodes file, and minimizer index

In addition to the graph, Giraffe also needs a distance index, a zipcodes file, and a minimizer index. These can be specified with options `-d` / `--dist-name`, `-z` / `--zipcode-name`, and `-m` / `--minimizer-name`. Specifying the index files explicitly is recommended in scripts and especially in production use.

If the indexes are not specified and the graph is named `graph.gbz` or `graph.giraffe.gbz`, Giraffe will guess the index names. For the distance index, it will check `graph.dist`. For the zipcodes file, it will try `graph.shortread.zipcodes` or `graph.longread.zipcodes` depending on the parameter preset, and then `graph.zipcodes`. For the minimizer index, it will try `graph.shortread.withzip.min` or `graph.longread.withzip.min` depending on the preset, then `graph.withzip.min`, and finally `graph.min`. If these files do not exist, Giraffe will try to build the indexes. This can cause issues when running multiple Giraffe jobs using the same indexes, or when the more specific/newer files are absent but older files are present. (For example, and old `.min` file without zipcode support might be picked up when what you really wanted was for Giraffe to build a new `.longread.withzip.min` file.)

The distance index is a memory-mapped file. **As of vg version 1.48.0**, the file will be opened in read+write mode by default. This can cause issues in HPC clusters and other distributed environments, where multiple computers try to access the same distance index file. To avoid this, make the file read-only or use a local copy of the file.

Giraffe relies on distance index annotations (in the form of zipcodes) in the minimizer index and its associated zipcodes file. Without them, mapping speed will be slow. Minimizer indexes built using `vg autoindex` always contain them. If you build the minimizer index manually using `vg minimizer`, you must specify the distance index using option `-d` / `--distance-index`, and you must also build the zipcodes file at the same time with `-z` / `--zipcode-name`.

### Reads

The reads can be either in FASTQ format (option `-f` / `--fastq-in`) or in GAM format (`-G` / `--gam-in`). A FASTQ file can be gzip-compressed.

By default, Giraffe does single-end mapping. For paired-end mapping, either specify two FASTQ files or use option `-i` / `--interleaved` with a single interleaved input file.

When doing paired-end mapping, Giraffe assumes that the first few thousand reads are a representative sample of the overall file and uses them for estimating fragment length distribution. If the input file is sorted, this does not work, and you must specify the fragment length distribution using options `--fragment-mean` and `--fragment-stdev`.
