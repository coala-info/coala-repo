# dascrubber

## Overview

DASCRUBBER is a suite of tools designed to "scrub" long-read sequencing data, primarily PacBio reads, to improve assembly quality. It identifies chimeras and unremoved adapters while filtering out segments where the error rate exceeds a specific threshold (typically 20%). The pipeline relies on alignment piles generated by `daligner` and requires that repeat-masking (via `DAMASKER`) has been applied. It transforms raw reads into a set of edited, high-quality sequences by moving through a specific sequence of coverage analysis, quality value assignment, and trimming.

## Pipeline Workflow

The scrubbing process must be executed in the following order. Parameters are often propagated via annotation tracks, so they do not always need to be repeated in every step.

1.  **DAScover**: Generates a histogram of coverage depth for trace point tiles.
2.  **DASqv**: Produces quality values (QV) for each trace point tile based on the coverage alignment matches.
3.  **DAStrim**: Defines the "good" and "bad" segments of reads based on QV thresholds.
4.  **DASpatch**: (Preliminary) Patches reads based on the trimming results.
5.  **DASedit**: (Preliminary) Produces the final edited database of scrubbed reads.

## Command Line Patterns

### Handling Database and Overlaps
*   **Source Argument**: The `<source>` or `<subject:db>` argument must always refer to the **entire database**.
*   **Overlaps Argument**: The `<overlaps:las>` argument can refer to specific block numbers (e.g., `DB.2.las`).
*   **Block Sequences**: Use the `@` symbol to process sequences of files:
    *   `DB.@`: Processes `DB.1.las`, `DB.2.las`, etc., until no file is found.
    *   `DB.@5`: Starts the sequence at block 5.
    *   `DB.@1-10`: Processes blocks 1 through 10 inclusive.

### Merging Tracks
Most modules produce block-level tracks (e.g., `.covr` or `.qual`). You must use `Catrack` to merge these into a single DB-wide track before proceeding to the next phase of the pipeline.
Example: After running `DAScover` on all blocks, run `Catrack <db> covr`.

## Tool-Specific Guidance

### DAScover
*   **HGAP Assembly**: Use the `-H<int>` option to only scrub reads longer than the specified integer. Shorter reads will still be used to help assess the longer "H-reads" but will not be scrubbed themselves.
*   **Masking**: Use `-m<track>` to specify repeat-masking tracks to ignore during coverage calculation.

### DASqv
*   **Threshold Assessment**: Use the `-v` option to view a histogram of quality values. This is critical for determining the appropriate `-g` (good) and `-b` (bad) thresholds for `DAStrim`.
*   **Default Thresholds**: By default, the system sets `-g` at the 80th percentile and `-b` at the 7th percentile of the QV distribution.

### DAStrim
*   **Input Requirement**: Requires a DB-wide `.qual` track produced by `DASqv` and merged by `Catrack`.
*   **Refining Results**: Adjust `-g` and `-b` based on the `DASqv -v` output if the default scrubbing is too aggressive or too lenient.

## Expert Tips
*   **Repeat Masking**: Never run DASCRUBBER without first employing repeat-masking in the `daligner` run. Failure to mask repeats will lead to incorrect coverage depth calculations and over-trimming of reads.
*   **Trace Spacing**: Ensure your `daligner` runs use a consistent `-s` (trace spacing) parameter, as `DASqv` relies on these tiles (default 100bp).
*   **Memory/Performance**: When working with large databases, process `.las` files block-by-block using the `@` syntax to manage memory effectively, then merge tracks.
