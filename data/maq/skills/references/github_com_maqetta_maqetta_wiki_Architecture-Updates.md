# Architecture Updates

[Jump to bottom](#wiki-pages-box)

·

## Current State

When we moved to using Orion as a base in M7, we were left with a **shim layer** of server code which acts as a facade between our old pre-Orion APIs and those of Orion. The shim layer is full of bugs and inefficient and should be removed.

Below are suggestions for future work to simplify and enhance our architecture and to take advantage of Orion as much as possible.

## Remove Server Shim Layer

Get rid of our old server APIs and the shim layer between those APIs and the Orion APIs (i.e. IStorage, IUser, etc). **Directly use the Orion APIs** as much as possible.

On the client side, make **requests to the Orion REST APIs** as much as possible. The client-side resource model modules at `davinci/model/resource`, which act as a write-through cache, should be tossed. Rather than fetching and caching the entire directory structure from root, resources and file lists should be accessed directly. If cached, they should be cached individually.

Since we use Orion as our workspace/project file store, we should be able to query the Orion REST APIs to get file info/contents. The big exception is with our *virtual directories/files*, such as `libs`, `themes` and `samples`. We may be able to create an **Orion filesystem plugin** to inject these virtual dirs/files. The advantage of this is that they would also show up in the Orion Navigator.

## Cleanup REST API

Our REST APIs are numerous and poorly thought out. We need to **cleanup and consolidate** the APIs to make them **more logical** and to **decrease network requests**. And we should properly use the GET, PUT, POST and DELETE operations.

## Server Architecture

Ensure server is as **stateless** as possible (except for caching or other performance enhancements, although those could be pushed out to a separate process [see Orion wiki linked below]). We should be able to **kill the server and restart without data loss**. The biggest issue here is if a user is editing a file -- I believe we currently fail silently when client tries to save working copies, so user does not know that his/her changes aren't being saved.

Ensure **mimimal startup time**. Most of this is out of our control, since much of the startup will be for Orion and Equinox (mostly the latter). But we need to test the Maqetta plugins.

Cleanup/standardize server **logging**. A server admin should be able to enable/disable logging on the fly.

## Security

Properly support **HTTPS**. The biggest obstacle at the moment is when users add a URL to a widget (such as to GridX) that isn't HTTPS; due to browser restrictions, this causes a security error. Need a way to handle that situation well.

Our REST APIs have custom code to do security checks, but they aren't standardized across all our requests. Should **standardize on Orion security APIs** (LoggedInUserFilter and AuthorizedUserFilter).

## Other

See [bug 404935](https://bugs.eclipse.org/bugs/show_bug.cgi?id=404935) and this [wiki page](http://wiki.eclipse.org/Orion/Plan/3.0/Server_scalability) for info/discussion of upcoming Orion server scalability changes.
