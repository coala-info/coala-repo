# Description of Outputs

[Jump to bottom](#wiki-pages-box)

Ryan Doughty edited this page Feb 17, 2026
·
[3 revisions](/treangenlab/bronko/wiki/Description-of-Outputs/_history)

bronko can produce a number of outputs from the `bronko map` command. Below is a description of each, and some tips on interpreting the output.

## Overview of outputs

In the output folder for the run (by default `bronko_output` in the location where the command was launched), there will be a number of files.

There will be one overview file labelled `bronko_overview.tsv`-

Then there will also be between 1-n files for each sample, where n will depend on the output formats requested:

* `[sample_name].vcf` (default, always present)
* `[sample_name].fa` (only with `--consensus`)
* `[sample_name].tsv` (only with `--pileup`)
* `[sample_name]_counts.txt` (only with `--keep-kmer-info`)

If `--alignment` is requested, there may also be files labelled `[reference_seq].mfa` containing alignments using that reference

## TSV Overview (`bronko_overview.tsv`)

bronko produces an overview output file with general statistics about each sample run. The expected columns in this tsv file are listed and described briefly below.

1. `filename`: the full path to each sample run
2. `selected_genome`: the genome in the database that was chosen as representative for that sample (note: if using one genome they will all be the same)
3. `num_major_variants`: the number of major variants (>=50% MAF) detected in that sample in comparison to the selected genome
4. `num_minor_variants`: the number of minor variants (<50% MAF) detected in that sample in comparison to the selected genome
5. `breadth_coverage`: percentage of bases in the selected reference that were covered (given as value between 0-1, ex 0.967 means 96.7% of bases covered)
6. `depth_coverage`: estimated average depth of coverage across the covered bases. Please note this will be an underestimate of the true sequencing depth.
7. `num_perfect_kmers`: the number of unique kmers that were mapped perfectly (all bases in the kmer match all bases in the reference)
8. `num_variant_kmers`: the number of unique kmers that were mapped with a single variant (1 base is different between kmer and reference)
9. `num_unmapped_kmers`: the number of unique kmers that had no mapping found in the database

Some brief notes that might be helpful when interpreting these outputs:

* For high-quality amplicon sequencing, we tend to like to see 95-99% breadth of coverage; otherwise, you may want to try to make sure there are more representative genomes in your database.
* The number of perfect kmers is limited by the size of the genome, so as long as coverage is sufficient, this should be pretty consistent
* Generally, we find that the number of variant kmers is highly correlated with the sequencing depth of the sample. If you find that you have super high sequencing depth (say > 10000x), you may want to play with the `--min-kmers` parameter and increase it to something more like 10. Please note that this may affect minor variant inference, though (as these kmers are useful in differentiating between signal and noise).
* Unmapped kmers are similarly not super informative for the most part, unless you are seeing lots of unmapped kmers

## VCF output (`[sample_name].vcf`)

Each .vcf file contains the variant calls for each sample. These use VCFv4.5 format and should be able to be interpreted as so.

Under INFO, we include information on the total depth of that position (DP), the estimated allele frequency (AF), the depths of the forward and reverse strands for both the reference and alt allele (DP4), and also the strands odd ratio that was calculated (SOR).

## Pileup output (`[sample_name].tsv`)

The pileup is a tsv format pileup containing the counts estimated by bronko for every position. It contains the following columns:

1. `reference`: the fasta header from the reference (in many cases this is the same, but can be multiple for viruses such as Influenza with multiple segments)
2. `index`: a global index for the position (1-based). For viruses with multiple segments, the index restarts at 1 when a new segment starts.
3. `ref`: the nucleotide at that position in the reference
4. `A`,`C`,`G`,`T`: the pileups counts along the forward strand (by default the reference strand)
5. `a`,`c`,`g`,`t`: the pileups counts along the reverse strand (by default reverse complement of the reference strand)

This can help give insight into the variant calls or identify regions where bronko may have missed things (due to indels or other reasons). Please also note that the depths are under-estimated (they represent a min bound on the depth at that position).

## Consensus output (`[sample_name].fa`)

The consensus sequence is often useful for other downstream applications outside of variant calling. Using `--consensus` will return a .fa file for each sample containing the consensus sequence. For any location where bronko was unable to find any sequence data in comparison to the reference, there are 'N's in that position. Given the approach, we often find that bronko has 'N's towards the end of each segment (usually 2-5 bases with reasonable quality sequencing).

## Alignments (`[reference_name`].mfa`)

The alignments output format is a .mfa format, which is the same format as .fa/.fasta, but instead of being a single sequence, each entry represents a genome. The first entry in this file is the reference, and each subsequent entry is a sample that was run where the representative genome was the given reference. The alignments contain every position across the set of genomes where at least 1 contained a mutation. These are concatenated in order to create the final sequence. This file can be used in downstream phylogenetics software to get phylogenetic trees for the samples.
