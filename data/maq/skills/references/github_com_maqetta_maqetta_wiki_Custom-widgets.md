NOTES ON CUSTOM WIDGETS:

### First, a quick overview of how built-in widgets work for comparison.

* Logic assumes that widget libraries have this file structure:

```
   <osgiproject>/WebContent/
                    <libname>/
                        ...library code... (e.g., dojo/{dojo/dijit/dojox})
                    metadata/
                          package.json
                          widgets/json
                          <packagename>... (e.g., dojo/dijit/dojox)
                                metadata files, with parallel structure to match library structure
                                (e.g., metadata/dijit/form/Button_oam.json)
```

* OSGi widget projects have `plugin.xml` file which defines an ajaxLibrary extension
* Maqetta logic ultimately uses the list of ajaxLibrary extensions to determine a list of built-in widget libraries
* Maqetta logic creates/maintains a `.settings/libs.settings` file in each user project which defines the virtual location of each of the built-in libraries (e.g., dojo defaults to `lib/dojo`)
* Same virtual location logic is put into module path definitions in `data-dojo-config` on runtime SCRIPT tag
* Maqetta server logic does a magic redirect in app and at runtime to map virtual locations to actual read-only widget library code
* Widget metadata folder (e.g., `davinci.dojo_1_8/WebContent/metadata`) gets maps inside page editor to package `maq-metadata-dojo`.
* Helper URLs must begin with `./` and the paths are relative to the metadata folder. e.g., `"helper":"./dijit/form/ButtonHelper"` maps to `maq-metadata-dojo/dijit/form/ButtonHelper.js`

### Custom widgets:

* Maqetta assumes that custom widgets within `lib/custom` will have this file structure:

```
       lib/custom/foo/
                      foo.js, foo.html, foo_widgets.json, foo_oam.json
 -or- (Note: OK to have nested folder structures)
       lib/custom/aaa/
                      bbb/
                          ccc.js, ccc_widgets.json, ccc_oam.json
```

* Each subfolder within `lib/custom` becomes its own module at run-time, and is added to the inner user iframe's list of loader packages and to source model's `data-dojo-config` list of packages e.g., assume a `lib/custom/foo`. Then `data-dojo-config` will have a "foo" package listed
* Save-as-widget exports a widget named "foo" into `lib/custom/foo` with four files in that folder: `foo/foo.js`, `foo/foo.html`, `foo/foo_widgets.json`, `foo/foo_oam.json`.
  + The resulting widget type will be `foo/foo`
* In page editor (app side of world), there will be a package named `maq-lib-custom-foo` (for loading of foo's helper files)
* Helper URLs must begin with `./` and the paths are relative to the metadata folder. e.g., assume a custom widget of type `aaa/bbb/ccc`, then `"helper":"./bbb/cccHelper"` maps to `maq-lib-custom-aaa/bbb/cccHelper.js` which means `lib/custom/aaa/bbb/cccHelper.js`
