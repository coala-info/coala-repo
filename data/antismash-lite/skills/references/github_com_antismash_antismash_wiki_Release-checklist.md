# How to create a new antiSMASH release

Internal documentation on how to create a new antiSMASH release.

## Local machine

* For a new major/minor release, branch off into a `<major>-<minor>-stable` branch.
* Push the branch to the main repository with the same HEAD as master
* Pin all dependencies to a specific version (or narrow range) and commit it
* Bump the version in `antismash/main.py` and commit it
* Create a pull request for the new branch (this tests the new environment)
* Update the local branch in order to tag the right patch.
* Use `git tag` to find the release tag of the last release
* Use `git shortlog <last-tag>.. > changes.txt` to create the changelog for the release notes file.
* Edit `changes.txt` to add more high-level release notes. Remember to add a link to the manuscript when releasing a new major release.
* Tag the new release using `git tag -s -F changes.txt <major>-<minor>-<patch>`
* Double-check everything looks like expected by using `git show <major>-<minor>-<patch>`. You should see the release notes, a GPG signature block, and your version bump commit.
* Push the new tag using `git push origin <tag-name>`

## GitHub

* Go to <https://github.com/antismash/antismash/releases> and click "Draft a new release" on the upper right corner.
* Put in your newly-created tag name into the tag textbox. You might need to select it from the drop-down of available tags again if GitHub claims the tag is new.
* Copy the content of your `changes.txt` file into the text box. I like to use triple-backticks around the detailed list of changes.

## Slack

* Tell the bot to create a new release tarball by running `!deploy antismash-release-tarball <major>-<minor>-<patch>`
* Update the current release number in `/glusterfs/downloads/dl-fancyindex-theme/footer_releases.html`

## Building docker containers

(see [antismash/containers](https://github.com/antismash/containers) for docker files)

### standalone-lite

* Update the version in the `VERSION` file.
* Update the `Dockerfile`, if needed.
* Run `./build.sh`, then `./publish.sh`.
* Copy the `download_antismash_databases_{docker,podman}` and `{docker,podman}-run_antismash-lite` scripts into `/data/downloads/releases/<current-release>`.

### webservice

* Update the version in the `VERSION` file.
* Update the `Dockerfile`, if needed.
* Run `./build.sh`, then `./publish.sh`.

### standalone

* Update the version in the `VERSION` file.
* Update the `Dockerfile`, if needed.
* Run `./build.sh`, then `./publish.sh`.
* Copy the `{docker,podman}-run_antismash-full` script into `/data/downloads/releases/<current-release>`.

## Slack

* Tell the bot to deploy the new stable image using `!deploy antismash-stable`
* If any dispatcher updates are required, deploy those after the image is deployed

## Documentation

* Add any new release notes and updates to the [antismash docs repository](https://github.com/antismash/documentation)
* Update any relevant sections of this Github wiki (e.g. rule keyword documentation when changes occur)

## Bonus

* Check the `asd_config.toml` files for antismash and fungismash are configured to run the right images.
* Announce new version on BlueSky
* Rebuild the bacterial (upload our copy of NC\_003888.3 to keep the SCO locus tags, RefSeq has broken those) and fungal (AAHF00000000.1) example outputs, and tell the bot to deploy them using `!deploy example <job-id>`
