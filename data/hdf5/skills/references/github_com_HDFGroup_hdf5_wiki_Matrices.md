## Bottom line up front (BLUF)

Every future concurrency PR has to answer one hard question: **which matrix row is being promoted, under what exact prerequisites, and what is the fallback?**

---

Distinguish between **safe because serialized** (`HDF5_ENABLE_THREADSAFE`) and **safe because truly concurrent** (`HDF5_ENABLE_CONCURRENCY`)

**Legend**

* **U** = unsupported / not a contract
* **S** = supported, but serialized by the library
* **C** = supported with true overlap
* **CF** = conditionally concurrent; requires explicit prerequisites and otherwise must fall back to **S**
* **O** = orthogonal process-scope feature, not part of the in-process thread contract

## User-facing Phase 0 matrix

| Scope | Preconditions | Phase 0 contract | Promotion target |
| --- | --- | --- | --- |
| Any API on a non-threadsafe build | More than one app thread may enter HDF5 | U | none |
| Any sequentially valid API on a threadsafe build | `HDF5_ENABLE_THREADSAFE` build | S | per-row below |
| `H5close` / library shutdown | Any HDF5-using thread may still run | U | stay U |
| Same physical file reopened independently | Separate opens of same physical file/path | U | stay U by default |
| Distinct physical files, read/read | Independent file IDs; no shared mutable helper objects | S | C early |
| Shared file ID, distinct objects, read/read | Same open handle graph; safe stack | S | CF early |
| Same dataset, disjoint simple hyperslabs, read/read | Native VOL, safe VFD/filters, no type conversion, no vlen, no app callbacks | S | CF early |
| Shared `H5P`/`H5S`/`H5T` objects, read-only | No concurrent mutation | S | C later |
| Same `H5P`/`H5S`/`H5T` object mutated while another op may use it | Any concurrent mutation | U | stay U unless redesign |
| Distinct physical files, write/write | Independent file IDs | S | C late |
| Shared file, different datasets, write/write | Disjointness not assumed by default | S | CF late |
| Same dataset, disjoint chunk write/write | Only when explicitly supported | S | CF late |
| Same dataset, overlapping write/write | Same chunk or overlapping logical region | S | stay S |
| Same object lifetime race | One thread may close while another uses same ID | U | stay U |

S means “the library supports the pattern only to the extent that the same sequence would be valid sequentially.” That is why overlapping writes can stay S forever without becoming C.

## Maintainer-facing companion matrix

Which package can be promoted from S to CF/C?

| Component class | Phase 0 default | First promotion gate | Hard rule |
| --- | --- | --- | --- |
| `H5`/`H5TS` global lifecycle | S | none in Phase 0 | shutdown races stay U |
| `H5E` / `H5CX` | S | thread-local correctness + callback re-entry tests | no shared mutable context |
| `H5I` / `H5P` | S | read-only sharing and refcount/lifetime tests | mutation of same object not concurrent |
| `H5VL` dispatch | S | connector stack classified safe | unknown stacks fall back to S or reject |
| VFD layer | S | small built-in whitelist only | unknown VFDs/plugins default S/U |
| `H5S` / `H5T` | S | per-op state, no shared mutable objects | shared mutable objects not concurrent |
| Native VOL read kernels | S | pre-materialized work units; no core locks across filters/callbacks | unsupported stacks fall back |
| Metadata cache / page buffer / chunk cache | S | explicit rearchitecture + deadlock tests | no accidental promotion |
| Writes / flush / file registry | S | selected disjoint cases only, late | overlapping mutation may remain S |

## CI/CD implications

Construct a machine-readable version of this for CI consumption. A row marked CF must say exactly what is checked at runtime:

* connector stack classified safe?
* VFD in whitelist?
* filters re-entrant?
* no app callback on path?
* shared objects read-only?
* same file via shared handle, not separate reopen?

If any prerequisite fails, the library must do one of two things only: fall back to S or reject cleanly. Never silently slide into undefined behavior.

## Principles

(Keep the matrix maintainable...)

Separate physical-file identity from ID/handle identity.

Separate process features from thread features.

Separate read-only sharing from mutation.

Keep some rows intentionally non-promotable. (Example: `H5close` with live HDF5 threads, close/use races on the same ID, and blind multiple opens of the same physical file are good candidates to remain U for a long time, maybe permanently.)
