## Does Autocycler include an assembly pipeline?

Autocycler itself does not include a full assembly pipeline. The ideal approach will depend on your computational environment and requirements. For example, some users may need to use job schedulers like [SLURM](https://slurm.schedmd.com/documentation.html), while others might use workflow managers such as [Nextflow](https://www.nextflow.io) or [Snakemake](https://snakemake.readthedocs.io/en/stable).

However, I have added the [`pipelines`](https://github.com/rrwick/Autocycler/tree/main/pipelines) directory to the Autocycler repository to gather such resources. It includes a [Bash-based pipeline created by myself](https://github.com/rrwick/Autocycler/tree/main/pipelines/Automated_Autocycler_Bash_script_by_Ryan_Wick) and pipelines submitted by others.

If you are optimising Autocycler assemblies or creating a pipeline, here are some things to keep in mind:

* The most time-consuming step in an Autocycler workflow is the creation of input assemblies, which can be carried out in parallel.
* The choice of input assemblers is up to you, see the next FAQ for some guidance.
* If your pipeline handles multiple isolates, I suggest running [Autocycler table](/rrwick/Autocycler/wiki/Autocycler-table) at the end to generate a summary table indicating how well each isolate assembled.
* You are welcome to create a pull request to add your creation to the [`pipelines`](https://github.com/rrwick/Autocycler/tree/main/pipelines) directory so others can use it. See that page for some basic contribution guidelines.

## Which assemblers should be used with Autocycler?

Autocycler is a consensus assembly tool, not an assembler itself, so you need to generate the input long-read assemblies using other tools. I have not done a systematic evaluation of exactly which combinations of assemblers work best with Autocycler, and I suspect the answer would be 'it depends' – some assemblers do better on certain taxa or types of sequences. So I recommend including at least a few different assemblers in your workflow.

Here are some of my thoughts on the available options:

* [Flye](https://github.com/mikolmogorov/Flye) is a good all-rounder and I recommend it for most workflows.
* [Canu](https://github.com/marbl/canu) produces accurate assemblies, but it can be very slow to run. Include it if you can spare the computational time. Skip it if you need a faster workflow.
* [Plassembler](https://github.com/gbouras13/plassembler) is slow to run and complicated to install (has many dependencies and needs a database), but it is the best tool at recovering small plasmids. I recommend including it if small plasmids matter to you.
* [miniasm](https://github.com/lh3/miniasm)(+[Minipolish](https://github.com/rrwick/Minipolish)), [Myloasm](https://github.com/bluenote-1577/myloasm), [Raven](https://github.com/lbcb-sci/raven) and [Redbean](https://github.com/ruanjue/wtdbg2) are all quite fast, often taking just 1 minute or so to run. If you need a quick pipeline, you can try using just these.
* Other options include [Hifiasm](https://github.com/chhylp123/hifiasm), [LJA](https://github.com/AntonBankevich/LJA), [metaMDBG](https://github.com/GaetanBenoitDev/metaMDBG), [NECAT](https://github.com/xiaochuanle/NECAT) and [NextDenovo](https://github.com/Nextomics/NextDenovo)(+[NextPolish](https://github.com/Nextomics/NextPolish)). None of these are critical, but they can be nice to include for more variety.
* All of the tools mentioned above can be run via [Autocycler helper](/rrwick/Autocycler/wiki/Autocycler-helper) for convenience.

## Polishing after Autocycler

Since Autocycler assemblies are long-read-only, they may still contain small-scale errors produced by systematic errors in the reads. A common example would be long homopolymers: if a genome contains `A×15` at a locus but most of the long reads erroneously have `A×14` at that locus, the assembly is likely to contain the `A×14` error.

If you are assembling Oxford Nanopore reads, performing long-read polishing with [Medaka](https://github.com/nanoporetech/medaka) can help. I recommend using the `--bacteria` option for the latest methylation-aware model. If you also have short reads, then I recommend using [Polypolish](https://github.com/rrwick/Polypolish) and [Pypolca](https://github.com/gbouras13/pypolca), both of which are conservative (unlikely to introduce new errors).

Further information about polishing is available in these papers:

* [Wick RR, Judd LM, Holt KE. Assembling the perfect bacterial genome using Oxford Nanopore and Illumina sequencing. PLOS Computational Biology. 2023. doi:10.1371/journal.pcbi.1010905.](https://doi.org/10.1371/journal.pcbi.1010905) and its [online tutorial](https://github.com/rrwick/Perfect-bacterial-genome-tutorial/wiki)
* [Bouras G, Judd LM, Edwards RA, Vreugde S, Stinear TP, Wick RR. How low can you go? Short-read polishing of Oxford Nanopore bacterial genome assemblies. Microbial Genomics. 2024. doi:10.1099/mgen.0.001254.](https://doi.org/10.1099/mgen.0.001254)

## How does Autocycler compare to Trycycler?

Autocycler was designed as a faster and automated successor to Trycycler. Both tools perform the same task: combining multiple alternative assemblies of the same genome into a clean consensus assembly. Here are some key differences:

* Trycycler is human-guided and almost always requires user intervention to complete. Autocycler is automated but still allows for human intervention when needed. For many genomes, Autocycler can complete without intervention.
* Both tools produce similar results, but Autocycler is generally easier to use.
* Trycycler is slower because it is written in Python and includes sequence alignment in some steps, which can be time-intensive. Autocycler is faster, being written in Rust and avoiding sequence alignment.

For most users, Autocycler is the better choice!

## How does Autocycler compare to Hybracter?

[Hybracter](https://github.com/gbouras13/hybracter) and Autocycler occupy different niches. Some key points:

* Hybracter is a full assembly pipeline that includes pre-assembly and post-assembly steps such as read QC, short-read polishing, and sequence reorientation. Autocycler focuses solely on long-read assembly and does not perform these additional steps.
* Hybracter uses a single long-read assembler (Flye), whereas Autocycler combines multiple long-read assemblies. This allows Autocycler to avoid Flye-specific errors in its consensus assemblies.
* Hybracter assemblies (`hybracter long`) can be used as input for Autocycler! This is particularly useful because Hybracter includes specialised logic to recover small plasmids, which other long-read assemblers often struggle with.
* While Autocycler often produces complete assemblies without intervention, some genomes may fail or require manual intervention to achieve the best results. If an absolutely no-intervention assembly approach is required, Hybracter is a better choice.
