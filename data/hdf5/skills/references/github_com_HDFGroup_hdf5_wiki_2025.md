# 2025-12-11

* **Facilitator/time-keeper:** Elena Pourmal
* **Note-taker/Editor:** AI/Elena Pourmal

# ***Agenda***

* Discuss [RFC: A Simple VFD Configuration Language](https://github.com/LifeboatLLC/HDF5-Encryption/blob/main/design_docs/RFC-VFD-Configuration-Language-2025-10-12.pdf) (John Mainzer, Lifeboat, LLC, 20 min)

# ***Minutes***

# Quick recap

The team discussed various configuration syntax options for VFDs, with a debate between using a Lisp-like syntax versus alternatives like YAML or JSON, ultimately deciding that while a standardized format could be useful, it shouldn't be mandatory and could
be developed as a third-party utility. The discussion covered implementation details including parser design, callback mechanisms, and security considerations, with Quincey and Scot emphasizing the importance of flexibility and avoiding reinventing existing
solutions. The group concluded that the parser would be made available as a third-party utility rather than being included in the HDF5 library, allowing VFD developers to choose their preferred parsing method while maintaining compatibility with the overall
system.

# Next steps

* [See Action Items and Conclusions section](%2A__Action-Items-and-Conclusions__%2A)

# Summary

**VFD Configuration Language RFC Discussion**

John presented his RFC on configuring a stack of VFDs using a single string in a language similar to Lisp. He explained the grammar, examples, and challenges in parsing and compiling the configuration. John discussed potential improvements, such as switching
to C-style comments and considering where to place the code, either in H5FD or a separate package. The group agreed to continue the discussion and extend the meeting if needed.

**HDF 5 Configuration Syntax Discussion**

The team discussed configuration syntax options for HDF5, with Scot and Quincey strongly advocating against using Lisp syntax, citing its obscurity and difficulty for users. They suggested alternatives like YAML or JSON, which Elena supported exploring. Scot also suggested to look into TOON and provided an example (cmp. with RFC):

`vfd_configs[3]{driver,page_size,buffer_size,mode}: page_buffer,4096,65536,LRU encryption,4096,32768,AES-GCM sec2,0,0,default`

Quincey
emphasized the need for flexibility in parsing, suggesting that each VFD could handle its own parsing rather than imposing a single standard. The group agreed that while a standardized configuration format could be useful, it should not be a requirement, and
third-party parsers could be developed for those who wish to use them.

**VFD Configuration Format Discussion**

The team discussed configuration formats for VFDs, with Quincey suggesting using JSON externally to the library to avoid maintaining a large component. Jordan explained that avoiding a standard format like YAML or JSON was preferable to potentially annoying
users, as different environments would likely use different formats anyway. Elena raised concerns about not considering current use cases, particularly for VFD Swimmer, which has limitations due to the inability to pass parameters between writer and reader.
The group agreed that third-party parsers could be used for VFD configuration, as long as it wasn't included in the main library.

**VFD Interface Expansion Discussion**

Quincey suggested expanding the VFD interface for better stacking mechanisms and interaction with lower layers, emphasizing the use of callbacks without mandating a specific language. Mainzer confirmed that the implementation was well underway and would be
made available for feedback. Scot raised concerns about security issues, error handling, and maintenance, to which Mainzer explained that the VFD would handle these by scanning parameters for errors and using a small, custom parser to minimize overhead.

**VFD Parser Implementation Discussion**

The group discussed implementing a parser for VFD (Vault File Driver) configuration strings, with Quincey and Scot expressing concerns about reinventing the wheel and preferring to use existing parsers. Quincey proposed adding callbacks to the VFD interface modeled on the vault connector system, which would allow for stackable VFDs and third-party parsers. The group agreed that while a standard format could be useful, it should not be mandatory and could evolve over time. They decided to make the parser available as a third-party utility rather than including it in the HDF5 library, giving VFD developers the flexibility to choose their preferred parsing method.

* THG doesn't want to use any standard language for configuring information to pass to VFDs and VOLs. Each connector should implement its own parser.
* Lifeboat will implement the proposed grammar and parser as part of VFDs they plan to contribute.
