# Genome Assembly with Pacbio Hifi reads and Trio data for phasing - VGP5

## Overview

## Overview

This workflow generates a high-quality, phased diploid genome assembly using PacBio HiFi long reads and parental Illumina data for trio-binning. As a core component of the [Vertebrate Genomes Project (VGP)](https://vertebrategenomesproject.org/) pipeline (VGP5), it is designed to be executed following the Trio k-mer Profiling workflow (VGP2). It utilizes [hifiasm](https://github.com/chhylp123/hifiasm) to leverage parental k-mers (hapmers) for accurate contig phasing, effectively separating maternal and paternal haplotypes.

The process begins by integrating HiFi reads with meryl and hapmer databases generated in previous steps. The workflow performs read trimming via Cutadapt and utilizes Genomescope parameters to inform the assembly process. The primary outputs include phased contigs for both haplotypes in FASTA and GFA formats, providing a comprehensive structural representation of the genome.

Extensive quality control is integrated into the pipeline to validate the assembly. It generates [BUSCO](https://busco.ezlab.org/) and Compleasm reports to assess gene content completeness, [Merqury](https://github.com/marbl/merqury) plots for k-mer based evaluation and phasing consistency, and [MultiQC](https://multiqc.info/) reports for read statistics. Detailed assembly metrics, including N50 statistics and size plots, are produced using gfastats and Bandage to ensure the final output meets VGP standards.

This workflow requires PacBio HiFi reads and parental Illumina data, typically provided as data collections for the child's reads and specific datasets for the maternal and paternal hapmer databases. Ensure you have the Meryl and Hapmer databases (`.meryldb`) and GenomeScope parameters generated from the VGP2 Trio k-mer Profiling workflow before execution. Key inputs include the taxonomic lineage for BUSCO/Compleasm and the homozygous read coverage, which can be manually refined or parsed from previous profiling steps. For a comprehensive guide on parameter settings and file preparation, refer to the `README.md`. You can also use `planemo workflow_job_init` to create a `job.yml` file for automated execution.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run Assembly-Hifi-Trio-phasing-VGP5.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run Assembly-Hifi-Trio-phasing-VGP5.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run Assembly-Hifi-Trio-phasing-VGP5.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init Assembly-Hifi-Trio-phasing-VGP5.ga -o job.yml`
- Lint: `planemo workflow_lint Assembly-Hifi-Trio-phasing-VGP5.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
