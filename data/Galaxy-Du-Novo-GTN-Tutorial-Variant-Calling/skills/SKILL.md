# du-novo-gtn-tutorial-variant-calling

## Overview

This Galaxy workflow is designed for high-sensitivity variant calling, specifically optimized for processing trimmed consensus reads (DCS or SSCS) generated by the Du Novo duplex sequencing pipeline. It automates the transition from raw consensus sequences to filtered, annotated variant calls, ensuring that the high-fidelity data produced by duplex sequencing is accurately interpreted.

The process begins by mapping paired-end reads to a reference genome using [BWA-MEM](https://toolshed.g2.bx.psu.edu/repos/devteam/bwa/bwa_mem/0.7.17.1). To improve calling accuracy near indels, the workflow utilizes [BamLeftAlign](https://toolshed.g2.bx.psu.edu/repos/devteam/freebayes/bamleftalign/1.1.0.46-0) to standardize the positioning of inserted or deleted bases before analysis.

Variant detection is performed by the [Naive Variant Caller (NVC)](https://toolshed.g2.bx.psu.edu/repos/blankenberg/naive_variant_caller/naive_variant_caller/0.0.4), which considers all aligned bases at every position. These results are then processed through a [Variant Annotator](https://toolshed.g2.bx.psu.edu/repos/nick/allele_counts/allele_counts_1/1.2) to provide detailed allele counts and metadata.

The final stage involves a filtering step to isolate high-confidence variants from background noise. The workflow outputs both the comprehensive annotated variant list and a refined, filtered dataset, making it a critical component of the [GTN](https://training.galaxyproject.org/) duplex sequencing tutorial series.

Ensure your input files are in fastqsanger or fastqsanger.gz format, specifically using the trimmed consensus reads (DCS or SSCS) generated from the Du Novo processing steps. While individual datasets work for single samples, organizing paired-end reads into a dataset collection is recommended for scaling this variant calling workflow efficiently across multiple samples. Refer to the accompanying README.md for comprehensive details on parameter settings and reference genome selection. You can automate the setup of these inputs using `planemo workflow_job_init` to generate a `job.yml` file.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run variant-calling.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run variant-calling.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run variant-calling.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init variant-calling.ga -o job.yml`
- Lint: `planemo workflow_lint variant-calling.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
