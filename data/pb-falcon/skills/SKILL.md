# pb-falcon

## Overview
The `pb-falcon` skill provides guidance on using the FALCON/Unzip suite, a set of tools designed for the assembly of long-read data into highly accurate, contiguous, and phased diploid genome assemblies. It utilizes the `pypeFLOW` engine to manage data processing pipelines, ensuring that intermediate steps are tracked and only necessary computations are re-run. This skill is essential for researchers working with large-scale genomic data who need to optimize assembly workflows and manage computational resources across distributed systems.

## Installation and Environment
The tool-suite is primarily distributed via Bioconda. Ensure your environment is correctly configured to avoid dependency conflicts.

- **Conda Installation**:
  `conda install bioconda::pb-falcon`
- **Environment Variables**:
  Use `PYPEFLOW_PRE` to inject specific environment settings or shell commands into the task scripts generated by the workflow engine.

## Workflow Management with pypeFLOW
`pb-falcon` relies on `pypeFLOW` to handle the directed acyclic graph (DAG) of assembly tasks.

- **Incremental Processing**: The engine tracks file dependencies. If a task fails or a file is updated, only the affected downstream tasks will be re-executed.
- **Task Execution**: Tasks can be executed as simple shell scripts or submitted to a cluster.
- **Process Watcher (pwatcher)**: The `pwatcher` component manages job submission and status tracking. It supports various backends including local execution, SLURM, and PBS.

## Configuration Best Practices
Assembly workflows are resource-intensive. Proper configuration of the `fc_run.cfg` (or equivalent configuration file) is critical.

- **Resource Allocation**:
  - Define `NPROC` to control the number of processors per task.
  - Define `MB` to specify memory requirements.
  - Note: Ensure your cluster configuration correctly interprets these values, as some versions may require manual overrides in the job submission string.
- **Job Types**:
  - Use `job_type=local` for small datasets or testing on a single high-core machine.
  - Use `job_type=slurm` or `job_type=pbs` for production-scale assemblies on a cluster.
- **Concurrency**:
  - Adjust `max_jobs` and `max_local_jobs` to prevent overwhelming the scheduler or the local file system's I/O capacity.

## Troubleshooting and Optimization
- **Job Completion**: If jobs are finished but not detected as complete, check the `pwatcher` logs. Network-based file systems can sometimes have latency in file visibility that affects dependency tracking.
- **Clean Up**: Use the built-in cleaning mechanisms to remove temporary files from failed runs before restarting to ensure a clean state.
- **Logging**: Review the `pypeflow.log` for detailed information on task transitions and dependency resolution.
