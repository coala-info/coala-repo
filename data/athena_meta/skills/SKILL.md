# athena_meta

## Overview

Athena is a read cloud assembler specifically designed for metagenomes. It functions by taking an initial set of "seed" contigs (generated by standard assemblers like metaSPAdes or IDBA-UD) and using the barcode information in read clouds to perform targeted local assemblies. This process fills gaps and extends contigs, resulting in a more contiguous and accurate metagenomic reconstruction. The tool operates through a structured pipeline: checking reads, performing subassemblies, and final OLC assembly.

## Installation and Environment

Athena requires a specific environment to function correctly:
- **Runtime**: Python 2.7 (not compatible with Python 3.x).
- **Dependencies**: `idba_ud` (specifically the version modified for read clouds), `samtools` (v1.3+), `bwa-mem`, and `flye` (v2.3.1).
- **Conda**: Available via `bioconda` as `athena_meta`.

## Input Requirements

Before running Athena, input data must be meticulously prepared:

1.  **Barcoded Reads**: Must be an uncompressed, interleaved paired-end FASTQ.
    - Reads must contain a `BC:Z:` or `BX:Z:` tag.
    - Barcodes must end with a dash and an integer (e.g., `-1`).
    - **Critical**: The FASTQ must be barcode-sorted so all reads with the same barcode are contiguous.
2.  **Seed Contigs**: A FASTA file of initial contigs, which must be indexed using `bwa index`.
3.  **Alignments**: A BAM file of barcoded reads aligned to the seed contigs.
    - Must be generated using `bwa mem -C -p` to ensure tags are passed through.
    - Must be position-sorted and indexed (`samtools index`).

## Configuration

Athena is driven by a `config.json` file. A minimal configuration includes:

```json
{
    "input_fqs": "/path/to/barcoded_sorted_reads.fastq",
    "ctgfasta_path": "/path/to/seed_contigs.fasta",
    "reads_ctg_bam_path": "/path/to/alignments_to_seeds.bam"
}
```

For high-performance computing environments, you can add `cluster_settings` to utilize schedulers like SLURM, SGE, Torque, or LSF via `ipython-cluster-helper`.

## Common CLI Patterns

### Validation and Testing
Check if all external dependencies are correctly installed and visible in the `$PATH`:
```bash
athena-meta --check_prereqs
```

Run a small-scale test assembly to verify the setup:
```bash
athena-meta --test
```

### Production Runs
Execute the assembly pipeline using a configuration file:
```bash
athena-meta --config path/to/config.json
```

### Resource Management
By default, Athena runs on a single thread. Use the `--threads` flag for parallel processing:
```bash
athena-meta --config config.json --threads 16
```
**Expert Tip**: Each thread requires approximately 4GB of RAM during the subassembly stage. Scale threads based on available system memory (e.g., 16 threads require ~64GB RAM).

### Troubleshooting
If input BAM or FASTQ files fail initial quality control but you wish to proceed:
```bash
athena-meta --config config.json --force_reads
```

## Output Structure
The results are placed relative to the directory containing the `config.json`:
- **Assembled Contigs**: `results/olc/athena.asm.fa`
- **Logs**: `logs/` (Check here first if the assembly fails).
