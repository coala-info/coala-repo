# UMI Grouping and Template‐Coordinate Sorting

# Overview

[GroupReadsByUmi](https://fulcrumgenomics.github.io/fgbio/tools/latest/GroupReadsByUmi.html) groups reads that appear to have come from the same original molecule by assigning the same Molecular Identifier `MI` tag. `GroupReadsByUmi` relies internally on the template-coordinate sort order.

The following describes:

1. How raw reads and templates are filtered prior to grouping.
2. How mapping coordinates and UMIs are used to identify and tag reads that originate from the same source molecule.
3. The template-coordinate sort order, which facilitates the efficient computation of (2).
4. How cell barcodes are used to partition reads by cell before grouping.

## Filtering reads and templates

A read is a single sequenced stretch of DNA and a template refers to all reads that share the same QNAME, typically a read pair (R1 and R2) from a single observed fragment.

| Concept | Definition | Example |
| --- | --- | --- |
| **Read** | A single sequenced strand (Read 1 or Read 2) | `@read123/1` |
| **Template** | The full DNA fragment sequenced, typically represented by both reads in a pair | `@read123` includes both /1 and /2 |

First the reads and templates are filtered to avoid the grouping of reads (and thus creation of consensus reads) from a single original molecule into two or more separate groups.

Individual reads are filtered if:

* Flagged as secondary and `--include-secondary` is false
* Flagged as supplementary and `--include-supplementary` is false

All reads for a template (read pair) are filtered if:

* All reads for the template are unmapped
* Any non-secondary, non-supplementary read has mapping quality < `'min-map-q'`
* Any UMI sequence contains one or more '`N`' bases
* `--min-umi-length` is specified and the UMI does not meet the length requirement

## Grouping of UMIs

The grouping of reads from their UMIs is performed by one of four strategies:

1. **identity**: only reads with identical UMI sequences are grouped together. Internally the identity method has a different implementation than the edit or adjacency methods. It's much simpler and certainly faster. The `edits` value should be set to zero. This approach can help with data exploration, but it should usually be avoided because errors in the UMI can split reads from the same original molecule into multiple groups.
2. **edit**: reads are clustered into groups such that each read within a group has at least one other read in the group with <= `edits` differences and there are inter-group pairings with <= `edits` differences. Effective when there are small numbers of reads per UMI, but breaks down at very high coverage of UMIs.
3. **adjacency**: a version of the directed adjacency method described in umi\_tools (<http://dx.doi.org/10.1101/051755>) that allows for errors between UMIs but only when there is a count gradient.
4. **paired**: similar to adjacency but for methods that produce a template such that a read with A-B is related to but not identical to a read with B-A. Expects the UMI sequences to be stored in a single SAM tag separated by a hyphen (e.g. `ACGT-CCGG`) and allows for one of the two UMIs to be absent (e.g. `ACGT-` or `-ACGT`). The molecular IDs produced have more structure than for single UMI strategies and are of the form `{base}`/{A|B}. E.g. two UMI pairs would be mapped as follows `AAAA-GGGG` -> 1/A, `GGGG-AAAA` -> 1/B. For more detail see [tracking reads through grouping.](https://github.com/fulcrumgenomics/fgbio/wiki/Developer-Note%3A-Tracking-Reads-through-Grouping-and-Duplex-Consensus-Calling#top-and-bottom-strand-for-raw-reads) This strategy is intended for duplex sequencing applications where the two ends of the molecule (seen in R1 and R2 respectively) each contain a UMI and the UMIs are applied in such a way during library construction that the top and bottom strand of the same molecule end up with the same UMIs (albeit inverted between R1 and R2).

Strategies **edit**, **adjacency**, and **paired** make use of the `--edits` parameter to control the matching of non-identical UMIs.

## Cell Barcode Support

When processing data with cell barcodes (e.g. single-cell or single-nuclei sequencing), the `--cell-tag` option (default: `CB`) controls how cell identity is incorporated into grouping. Reads at the same genomic coordinates are partitioned by cell barcode *before* UMI assignment, so reads from different cells will never be grouped together even if they share a UMI and mapping position. No correction is performed on the cell barcode itself; it is treated as a known/fixed value.

This means you do not need to split your BAM by cell barcode before running the consensus calling pipeline. Simply ensure your reads have cell barcodes in the appropriate tag and pass `--cell-tag` through each step:

```
fgbio GroupReadsByUmi --cell-tag CB ...
fgbio CallMolecularConsensusReads --cell-tag CB ...
```

The consensus callers (`CallMolecularConsensusReads`, `CallDuplexConsensusReads`) validate that all source reads in a group share the same cell barcode and propagate it to the output consensus read.

## Template-Coordinate Sorting

If the input is not already sorted in template-coordinate order (SO:unsorted GO:query SS:unsorted:template-coordinate), `GroupReadsByUmi` will internally sort the reads before processing.

To avoid this extra sorting step, it is recommended to pre-sort the input using [samtools sort](https://www.htslib.org/doc/samtools-sort.html), note that the `MI` tag is part of the samtools sort key but in this case will not yet be present:

`samtools sort --template-coordinate --threads $(nproc)`

Template-coordinate order sorts reads by:

1. the earlier unclipped 5' coordinate of the read pair
2. the higher unclipped 5' coordinate of the read pair
3. strand orientation
4. the cellular barcode (CB tag, length then value)
5. the molecular identifier (MI tag if present, length then value)
6. read name
7. library (from read group)
8. if R1 has the lower coordinates of the pair

Reads assigned to templates and grouped together by `GroupReadsByUmi`, having the same MI, will also have the same outer start/stop. Because the 5' coordinate is strand-aware (based on read orientation) the reads from opposite strands, even with the same UMI and position, will not be in the same template and not grouped together.

Please see the [Calling Consensus Reads](https://github.com/fulcrumgenomics/fgbio/wiki/Calling-Consensus-Reads) and [`CallDuplexConsensusReads`](https://github.com/fulcrumgenomics/fgbio/wiki/Calling-Duplex-Consensus-Reads) tools and the [Best Practices Consensus Pipeline](https://github.com/fulcrumgenomics/fgbio/blob/main/docs/best-practice-consensus-pipeline.md) for additional information.
