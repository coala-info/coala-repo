This allows a user to setup a directory tree for some software with directories &lt;root&gt;/bin and &lt;root&gt;/lib , and then the above example will allow the use of modules in the lib directory without knowing where the software tree is installed. If perl is invoked using the -e option or the Perl script is read from STDIN , then FindBin sets both $Bin and $RealBin to the current directory. EXPORTABLE VARIABLES $Bin or $Dir Path to the bin directory from where script was invoked $Script Basename of the script from which perl was invoked $RealBin or $RealDir $Bin with all links resolved $RealScript $Script with all links resolved You can also use the ALL tag to export all of the above variables together: use FindBin &#39;:ALL&#39;; KNOWN ISSUES If there are two modules using FindBin from different directories under the same interpreter, this won&#39;t work. Since FindBin uses a BEGIN block, it&#39;ll be executed only once, and only the first caller will get it right. This is a problem under mod_perl and other persistent Perl environments, where you shouldn&#39;t use this module. Which also means that you should avoid using FindBin in modules that you plan to put on CPAN. Call the again function to make sure that FindBin will work: use FindBin; FindBin::again(); # or FindBin-&gt;again; In former versions of FindBin there was no again function. The workaround was to force the BEGIN block to be executed again: delete $INC{&#39;FindBin.pm&#39;}; require FindBin; AUTHORS FindBin is supported as part of the core perl distribution.
