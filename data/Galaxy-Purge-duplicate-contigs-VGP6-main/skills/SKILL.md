# Purge duplicate contigs from a diploid assembly VGP6

## Overview

This workflow represents the sixth stage of the [Vertebrate Genomes Project (VGP)](https://vertebrategenomesproject.org/) assembly pipeline. It is specifically designed to refine diploid assemblies by identifying and removing haplotypic or overlap duplications using [purge_dups](https://github.com/dfguan/purge_dups). It is intended for use after initial contigging steps (VGP workflows 3, 4, or 5) to ensure the primary and alternate assemblies are correctly partitioned.

The process follows a sequential logic: redundant contigs are first purged from the primary assembly (hap1) and reassigned to the alternate assembly (hap2). Subsequently, the alternate assembly undergoes its own round of purging to ensure all duplications are resolved. This dual-purging approach is ideal for diploid genomes where both haplotypes are represented; for cases requiring only a single assembly purge, the VGP6b variant is recommended.

To run the pipeline, users provide trimmed HiFi long reads, the initial primary and alternate FASTA files, and k-mer profiling data (Meryl database and GenomeScope parameters). The workflow integrates several [VGP_curated](https://github.com/VGP/vgp-assembly) tools to generate a comprehensive suite of quality control outputs, including [BUSCO](https://busco.ezlab.org/) and [compleasm](https://github.com/huangnengCSU/compleasm) reports, [Merqury](https://github.com/marbl/merqury) k-mer evaluations, and detailed assembly statistics such as Nx and size plots.

Detailed information regarding input formatting and parameter settings can be found in the README.md in the Files section.

Ensure you have the trimmed HiFi long reads available as a data collection, alongside the primary and alternate assemblies in FASTA format. You will also need the Meryl database and GenomeScope model parameters generated during previous k-mer profiling steps. Key text inputs include the estimated genome size and the specific BUSCO lineage database required for ortholog evaluation. For a comprehensive guide on parameter settings and file preparation, refer to the README.md. You can use `planemo workflow_job_init` to generate a `job.yml` file for automated execution.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run Purge-duplicate-contigs-VGP6.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run Purge-duplicate-contigs-VGP6.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run Purge-duplicate-contigs-VGP6.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init Purge-duplicate-contigs-VGP6.ga -o job.yml`
- Lint: `planemo workflow_lint Purge-duplicate-contigs-VGP6.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.
