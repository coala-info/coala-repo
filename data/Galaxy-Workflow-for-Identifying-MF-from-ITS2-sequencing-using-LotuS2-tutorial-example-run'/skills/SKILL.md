# Workflow for Identifying MF from ITS2 sequencing using LotuS2 - tutorial example run'

## Overview

This workflow provides a streamlined pipeline for identifying fungal communities from ITS2 paired-end sequencing data using the [LotuS2](https://github.com/hildebrandlab/LotuS2) tool. Designed for applications in ecology and metagenomics, it follows established [Galaxy Training Network (GTN)](https://training.galaxyproject.org/) methodologies to process raw sequence data into taxonomically classified outputs.

The pipeline requires three primary inputs: a dataset collection of paired-end sequences, a mapping file (TSV), and an `sdm` options configuration file. These inputs are processed by the LotuS2 tool, which performs sequence demultiplexing, quality filtering, and clustering to identify the fungi present within the samples.

Upon completion, the workflow generates several essential outputs for downstream analysis, including an OTU table, an OTU FASTA file, and an OTU phylogenetic tree. A comprehensive ZIP archive is also produced, containing the complete set of results generated by the LotuS2 engine. This workflow is released under the [MIT license](https://opensource.org/licenses/MIT).

## Inputs and data preparation

| Step | Label | Type | Notes |
| --- | --- | --- | --- |
| 0 | Mapping tsv | data_input | Mapping file needed by LotuS2 to link sample metadata to a pair of fastq files |
| 1 | sdm Options txt | data_input | SDM (sequence demultiplexer tool) options needed for fungal ITS illumina paired-end sequencing |
| 2 | Input Dataset Collection | data_collection_input | List of paired fastq files needed to run LotuS2. These files MUST match the files in the Mapping File |


Ensure your fungal ITS2 paired-end sequences are organized into a paired dataset collection, while the mapping file and sdm options must be provided as individual TSV and TXT files respectively. Verify that the mapping file headers strictly follow LotuS2 requirements to avoid execution errors during the OTU clustering process. For comprehensive instructions on formatting these metadata files and configuring tool parameters, refer to the accompanying README.md. You can streamline the configuration of these inputs by using `planemo workflow_job_init` to generate a template `job.yml` file.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run lotus2-fungi-workflow.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run lotus2-fungi-workflow.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run lotus2-fungi-workflow.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init lotus2-fungi-workflow.ga -o job.yml`
- Lint: `planemo workflow_lint lotus2-fungi-workflow.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
