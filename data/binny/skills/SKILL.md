# binny

## Overview

binny is an automated binning algorithm designed to recover high-quality Metagenome-Assembled Genomes (MAGs) from metagenomic datasets. It functions as a Snakemake-based workflow that integrates contig sequences with abundance information to provide a streamlined pipeline for genome reconstruction. This skill provides guidance on using the binny executable wrapper to manage environment initialization, dry runs, and execution across local or cluster environments.

## Installation and Initialization

Before running binning workflows, the environment and required databases (such as CheckM marker genes) must be prepared.

*   **Conda Installation**: Install via bioconda using `conda install bioconda::binny`.
*   **Environment Setup**: If working from a cloned repository, initialize the conda environments and databases using the `-i` flag:
    `./binny -i config/config.init.yaml`
*   **Database Note**: binny automatically handles the download and preparation of CheckM marker gene databases during initialization.

## Execution Patterns

The `binny` executable is a wrapper that simplifies Snakemake execution. It requires a configuration file as the final argument.

### Validation and Testing
Always perform a dry run before committing resources to a full metagenomic binning task. This validates input paths and sample formatting.
*   **Dry Run**: `binny -d config.yaml`
*   **Test Run**: Use the provided test configuration to verify the installation:
    `./binny -l -n "TESTRUN" -r config/config.test.yaml`

### Local and Frontend Execution
For smaller datasets or when running on a powerful workstation:
*   **Standard Local Run**: `binny -l -r config.yaml`
*   **Background Execution (Tmux)**: Use `-f` to run in a tmux session on the frontend, which prevents the process from dying if your connection drops:
    `binny -f -n "MY_RUN" -r config.yaml`

### Cluster Execution
binny is designed to scale to HPC environments using schedulers like Slurm or SGE.
*   **Cluster Submission**: `binny -c -r config.yaml`
*   **Cluster with Tmux**: Use `-c -x` to run the Snakemake master process in a tmux session while it submits individual jobs to the cluster.

## Command Line Arguments

| Flag | Description |
| :--- | :--- |
| `-d` | **Dryrun**: Check logic and file paths without executing. |
| `-l` | **Local**: Run in the current terminal session. |
| `-c` | **Cluster**: Submit jobs to a cluster scheduler. |
| `-x` | **Tmux**: Run the master process inside a tmux session. |
| `-r` | **Report**: Generate a report upon completion. |
| `-n [NAME]` | **Name**: Assign a specific name to the job (no spaces allowed). |
| `-t [INT]` | **Threads**: Set the maximum number of CPUs to use simultaneously. |
| `-u` | **Unlock**: Unlock the output directory if a previous run was interrupted. |

## Expert Tips and Best Practices

*   **Input Requirements**: Ensure you have your assembled contigs in FASTA format and either read alignments in BAM format or a tab-separated depth file.
*   **Resource Management**: When running on a cluster, use the `-t` flag to limit concurrent CPU usage if your environment has strict quotas. The default for cluster runs is 50.
*   **Monitoring Progress**: If using the tmux flag, you can reattach to the session using `tmux ls` and `tmux attach`. Alternatively, monitor the standard error log generated by the naming flag: `tail -f NAME_XXXXXXXXXX.stderr`.
*   **Output Structure**: A successful run populates the `bins/` subdirectory within your designated output folder with FASTA files, each representing a single recovered bin.
