# Command line

```
[experimental] Process a taxonomy and lookup lineages. Called as `blobtk taxonomy`

Usage: blobtk taxonomy [OPTIONS]

Options:
  -t, --taxdump <PATH>
          Path to backbone taxonomy file/directory
  -f, --taxonomy-format <TAXONOMY_FORMAT>
          Format of taxonomy file [possible values: ncbi, gbif, ena, ott, genome-hubs]
  -r, --root-id <ROOT_TAXON_ID>
          Root taxon/taxa for filtered taxonomy
  -l, --leaf-id <LEAF_TAXON_ID>
          Leaf taxon/taxa for filtered taxonomy
  -b, --base-id <BASE_TAXON_ID>
          Base taxon for filtered taxonomy lineages
  -O, --taxdump-out <OUT>
          Path to output filtered backbone taxonomy
  -c, --config <CONFIG_FILE>
          Path to YAML format config file
  -x, --xref-label <XREF_LABEL>
          Label to use when setting as xref
  -g, --genomehubs_files <GENOMEHUBS_FILES>
          Files to match to taxIDs - Experimental
      --api
          [experimental] Start the taxonomy API server instead of running a one-off merge/output
  -p, --port <PORT>
          Port to run the API server on (if --api is set) [default: 3000]
  -h, --help
          Print help (see more with '--help')
```

## Filter a taxonomy

Specify `-r/--root-id` to select only that taxon and its descendants from the full taxonomy.

Optionally specify `-b/--base-id` to extend the lineage past the root taxon to a more ancestral taxon.

### NCBI taxonomy

subset for taxID 9608 (Canidae):

```
blobtk taxonomy -t taxdump/ncbi -f ncbi -r 9608 -O taxdump/canidae
```

subset for taxID 9608 (Canidae) and taxID 3744 (Rosales) with lineages up to taxID 2759 (Eukaryota):

```
blobtk taxonomy -t taxdump/ncbi -f ncbi -r 9608 -r 3744 -b 2759 -O taxdump/canidae_rosales
```

### GBIF backbone taxonomy

subset for taxID 9701 (Canidae):

```
blobtk taxonomy -t taxdump/gbif/simple.txt -f gbif -r 9701 -O taxdump/canidae_gbif
```

subset for taxID 9701 (Canidae) and taxID 691 (Rosales) with lineages up to taxID 0 (Eukaryota is not present so an artificial root taxon is created):

```
blobtk taxonomy -t taxdump/gbif/simple.txt -f gbif -r 9701 -r 691 -b 0 -O taxdump/canidae_rosales_gbif
```

## Use a config file

All command line options and several additional options can be set using a YAML format config file by specifying `-c/--config`.

If the following config file is saved as `example.config.yaml`

```
path: taxdump/ncbi
out: taxdump/combined/canidae_rosales
taxonomy_format: ncbi
root_taxon_id:
  - 9608
  - 3744
base_taxon_id: 2759
name_classes:
  - scientific name
  - synonym
taxonomies:
  - path: taxdump/gbif/simple.txt
    taxonomy_format: gbif
    root_taxon_id:
      - root
    base_taxon_id: root
    name_classes:
      - scientific name
    xref_label: gbif
```

it can be passed to blobtk taxonomy as

```
blobtk taxonomy -c config.yaml
```

Using a config file in this way allows parameters such as file location and xref label to be set on a per-file basis, which is useful when merging taxonomies, adding new taxa and matching data files to a taxonomy.

## Taxonomy API

The parsed taxonomy and name lookup can be served as an API to allow for reuse by external processes. To start the taxonomy API, define a config file like:

config\_api.yaml

```
path: taxdump/ncbi
taxonomy_format: ncbi
root_taxon_id:
  - 1
base_taxon_id: 1
name_classes:
  - scientific name
  - synonym
xref_label: ncbi
```

### Start the API

To start the taxonomy API, run

```
blobtk taxonomy --api -c config_api.yaml --port 3000
```

### Check API status

API status is reported by the `/status` endpoint. This should return `initializing` while the taxonomy is being parsed and `ready` once the lookup table has been built.

```
curl http://localhost:3001/status
{"status":"initializing"}

curl http://localhost:3001/status
{"status":"ready"}
```

### Validate a GenomeHubs file pair

A GenomeHubs YAML/TSV file pair can be validated against the `/validate` endpoint. Validation includes name matching and exception file creation. The file pair to be validated must be present on a disk that can be directly accessed by the API.

```
curl -X POST http://localhost:3001/validate \
  -H "Content-Type: application/json" \
  -d '{"file_path": "ATTR_assembly.types.yaml"}'
```

The response will either be OK:

```
{"valid":true,"message":"OK"}
```

or an error message:

```
{"valid":false,"message":"SerdeError(\"/Users/rchallis/projects/genomehubs/goat-data/sources/assembly-data/ATTR_assembly.types.yaml attributes.qv_score.constraint: unknown field `type`, expected one of `enum`, `len`, `max`, `min` at line 305 column 7\")"}
```

This status reflects whether the YAML itself adheres to the schema. Individual rows/values may still fail validation and these will be shown in an `exceptions` directory alongside the YAML file being validated.

# Fetch source taxonomy files

The following commands can be used to fetch the source taxonomy files used in the examples above.

## NCBI taxonomy taxdump

```
mkdir -p taxdump/ncbi &&
curl -L https://ftp.ncbi.nlm.nih.gov/pub/taxonomy/taxdump.tar.gz > taxdump/ncbi/taxdump.tar.gz &&
tar xf taxdump/ncbi/taxdump.tar.gz -C taxdump/ncbi
```

## GBIF backbone taxonomy

```
mkdir -p taxdump/gbif &&
curl -L https://hosted-datasets.gbif.org/datasets/backbone/current/simple.txt.gz | gunzip -c > taxdump/gbif/simple.txt
```
