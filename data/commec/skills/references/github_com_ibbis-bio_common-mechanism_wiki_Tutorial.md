# Tutorial

[Jump to bottom](#wiki-pages-box)

This tutorial will guide you through your first DNA sequence screening using commec. We'll start with lightweight screening using `--skip-tx` and then explore more advanced features.

1. [Quickstart with skip taxonomy](#quickstart-with-skip-taxonomy)
2. [Using YAML to configure commec screen](#using-yaml-to-configure-commec-screen)
3. [Interpreting commec screen results](#interpreting-commec-screen-results)
4. [Interpreting commec flag outputs](#interpreting-commec-flag-outputs)

## Quickstart with skip taxonomy

For most users, this is the fastest way to get started and ensure the package has installed correctly:

```
# Download core databases using commec setup (~2GB)
commec setup
```

You should then create a config YAML file pointing to your databases (you can add any other parameters you will want by default). We recommend running with 4 threads:

```
cat <<EOF > config.yaml
base_paths:
  default: /mnt/data/home/ec2-user/cm-dbs/
threads: 4
EOF
```

You can now run a truncated commec screen on an input FASTA of your choice with:

```
commec screen --skip-tx input.fasta -y config.yaml
```

## Using YAML to configure commec screen

We recommend configuring your `commec screen` runs by providing a YAML file:

```
commec screen input.fasta -y config.yaml
```

By default, the `commec` package will use the [screen-default-config.yaml](https://github.com/ibbis-bio/common-mechanism/blob/main/commec/screen-default-config.yaml) included in the package to provide defaults for all `screen` parameters not provided on the command line. This file has the following contents:

```
base_paths:
  default: commec-dbs/
databases:
  biorisk:
    path: '{default}biorisk/biorisk.hmm'
    taxids: "{default}biorisk/reg_taxids.txt"
    annotations: '{default}biorisk/biorisk_annotations.csv'
  regulated_protein:
    blast:
      path: '{default}nr_blast/nr'
    diamond:
      path: '{default}nr_dmnd/nr.dmnd'
  regulated_nt:
    path: '{default}nt_blast/core_nt'
  low_concern:
    rna:
      path: '{default}low_concern/rna/benign.cm'
    dna:
      path: '{default}low_concern/dna/benign.fasta'
    protein:
      path: '{default}low_concern/protein/benign.hmm'
    taxids: "{default}low_concern/vax_taxids.txt"
    annotations: '{default}low_concern/low_concern_annotations.tsv'
  taxonomy:
      path: "{default}taxonomy/"
threads: 1
diamond_jobs: null
do_cleanup: False
force: False
skip_taxonomy_search: False
protein_search_tool: 'blastx'
resume: False
skip_nt_search: False
verbose: False
```

### Pointing to locally-installed databases directory

Most users will not have downloaded databases to `commec-dbs` and will want to override the default databases directory. If you have placed all of your databases in a single directory, you may be able to only override the default base path, for example by providing the following override yaml in the command line:

#### default\_override.yaml

```
base_paths:
  default: /mnt/cloud-user/data/commec-dbs/
```

Overriding `default` gives you the same behaviour as providing that directory to the `-d/--databases` argument:

```
# These commands will behave identically
commec screen --skip-tx input.fasta -y default_override.yaml
commec screen --skip-tx input.fasta -d /mnt/cloud-user/data/commec-dbs/
```

### Pointing to locally-installed databases in multiple locations

More complex overrides are possible. For example, if you are on a shared compute cluster that already has NCBI databases installed, and you know that you want your runs to use up to 4 threads, you might use the following configuration:

#### db\_config.yaml

```
base_paths:
  default: /mnt/cloud-user/data/commec-dbs/
  ncbi: /mnt/share/bioinformatics/ncbi/blast/
databases:
  regulated_nt:
    path: '{ncbi}/nt_core/nt'
  regulated_protein:
    blast:
      path: '{ncbi}/nr/nr'
threads: 4
```

You can combine YAML configuration and command-line flags. For example, if you want to debug a run using the above configuration, and test whether it’s faster with 8 threads, you might run:

```
commec screen input.fasta -y db_config.yaml --threads 8 -v
```

### Warning: YAML hierarchy matters

Configuration is loaded based on the hierarchical structure in [screen-default-config.yaml](https://github.com/ibbis-bio/common-mechanism/blob/main/commec/screen-default-config.yaml). Your overrides must follow the same hierarchical structure; values outside of this structure will not be used. Contrast:

#### db\_config.yaml

```
base_paths:
  default: /mnt/cloud-user/data/commec-dbs/
  ncbi: /mnt/share/bioinformatics/ncbi/blast/
databases:
  regulated_nt:
    path: '{ncbi}/nt_core/nt'
```

resulting in the `regulated_nt` path being set to */mnt/share/bioinformatics/ncbi/blast/nt\_core/nt*, but this, because of the missing top-level `databases` field, will not:

#### error\_in\_formatting\_confg.yaml

```
base_paths:
  default: /mnt/cloud-user/data/commec-dbs/
  ncbi: /mnt/share/bioinformatics/ncbi/blast/
  regulated_nt:
    path: '{ncbi}/nt_core/nt`
```

and `regulated_nt` will be set to */mnt/cloud-user/data/commec-dbs/nt\_blast/nt* (the default `path`, using the user-specified `default` base\_path).

## Interpreting commec screen results

The common mechanism repository includes a directory of [`example_data/`](https://github.com/ibbis-bio/common-mechanism/tree/main/example_data).

The examples include queries that illustrate different possible screening outcomes, and the directory contains the results of results of running `commec screen` on the mutli-query `.fasta` containing the examples.

We can interrogate the outcome of commec screen for each example through its entries in the [`_summary.html`](https://github.com/ibbis-bio/common-mechanism/blob/main/example_data/commec-examples_summary.html) graphical report, as well the [`output.json`](https://github.com/ibbis-bio/common-mechanism/blob/main/example_data/commec-examples.output.json) data file.

### Example output directory structure

The output file structure of commec screen, here using `-o /output/` and `example.fasta` creates the following directory structure:

```
output/                               # Parent directory
├── example_input/                    # Inputs to each screen step.
│   ├── example.cleaned.fasta         # a copy of the input fasta
│   ├── example.noncoding.fasta       # identified non-coding regions for step 3
│   ├── example.faa                   # 6-frame translations
│   └── example_config.yaml           # A copy of the .yaml used.
│
├── example_output/                   # Raw output data directory
│   ├── example.biorisk.hmmscan       # Raw Biorisk output, Step 1
│   ├── example.nr.blastx             # Raw Protein Taxonomy output
│   ├── example.nt.blastn             # Raw Nucleotide Taxonomy output
│   ├── example.lowconcern.blastn     # Raw Low concern dna output
│   ├── example.lowconcern.cmscan     # Raw low concern rna output
│   └── example.lowconcern.hmmscan    # Raw low concern protein output
│
├── example.output.json               # output .json summary file.
├── example_summary.html              # graphical summary .html file
└── example.screen.log                # text log for commec screen
```

### Example 1: BBa\_K5108009\_creA\_

This is a [composite DNA part](https://parts.igem.org/Part%3ABBa_K5108009) developed by 2024 iGEM team Toulouse-INSA-UPS for space exploration applications. It is an artificial operon composed of four basic parts: creatinase and creatinine amidohydrolase ORFs (creA BBa K5108003, crnA BBa K5108004) and two RBS (BBa K5108006, BBaK5108007) enabling their expression in the plant growth-promoting rhizobacteria, *Pseudomonas fluorescens*, enabling the metabolization of creatinine by this organism.

![BBa_K5108009_creA_ HTML Report](./graphics/BBa_k5108009_creA_.png)
