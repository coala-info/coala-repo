# Detection rules

## A general introduction

Protoclusters in antiSMASH are found using a rule based method. These rules define patterns which are required within co-located genes.

Rules are defined with a domain-specific language, for which the full grammar is described [here](https://github.com/antismash/antismash/blob/master/antismash/common/hmm_rule_parser/rule_parser.py).

Current rules are found [here](https://github.com/antismash/antismash/blob/master/antismash/detection/hmm_detection/cluster_rules), though if using a variant of antiSMASH they will found in the repository of that variant.

A reasonably complicated rule is the `transAT-PKS` rule, which is shown here:

```
RULE transAT-PKS
    CATEGORY PKS
    DESCRIPTION Trans-AT polyketide
    EXAMPLE NCBI CP006259.1 382565-499230 kirromycin
    EXAMPLE NCBI CP001511.1 5000-30000 toblerol
    CUTOFF 20
    NEIGHBOURHOOD 20
    CONDITIONS cds(PKS_AT and not ANY_KS)
               and (
                cds(ATd and ANY_KS)
                # handle CP001511 5k-30k as per https://doi.org/10.1002/ange.201709056
                or cds(tra_KS and PP-binding)
               )
    EXTENDERS cds(ANY_KS and not PKS_AT)
```

Not all of these sections are required in a rule, and there are more sections available. Included are the description and example genomes, which is important when creating and/or modifying rules as it provides a good test set. It also includes comments to help understand why a particular section exists.

The `CONDITIONS` field is the critical portion of the rule. For details on the construction of these conditions, see [here](#detection-rule-conditions). These defines the conditions under which a protocluster can be formed, described in terms of HMM profiles (for full genes or domains within genes) or dynamic, code-based detection specific kinds of patterns that cannot be captured by HMM profiles. This example, specifically, requires that a CDS/gene exists that has within it a `PKS_AT` and no `ANY_KS` (which is an alias, defined earlier in the file, for a variety of possibilities). It also requires that one of the two conditions separated by an `or` is present within the cutoff distance.

The `CUTOFF` field is the maximal range (in kilobases) that the algorithm will search up- and down-stream of a CDS/gene that satisfies at least one of the conditions specified. If all other portions of the `CONDITIONS` field can be satisfied by genes within that cutoff, a protocluster will be formed.

The `NEIGHBOURHOOD` field is used to provide genomic context for the protocluster, as often there are genes nearby that don't match the rule profiles themselves (e.g. tailoring enzymes).

The `EXTENDERS` field is similar to the `CONDITIONS` field and is used to extend the borders of the cluster core in cases where the extension conditions are less strict than the initial conditions for forming the cluster.

## Creating new rules

The first step in creating a rule is finding the components of the new cluster type that will form the `CONDITIONS`. After these components are identified, they will need to have a mechanism of detection. For this there's two methods; HMM profiles or dynamic, code-based profiles.

### Creating and using a HMM profile

HMM profiles are built from a selection of sequences. The process to select sequences won't be covered here, but be careful to avoid overfitting. The building of a profile itself is covered in the [HMMer User's Guide](http://eddylab.org/software/hmmer/Userguide.pdf) and requires the installation of the HMMer toolset. The simplified instructions are to use the command `hmmbuild <hmmfile> <input_sequence_file>`. For compatibility with the other profiles used in detection, ensure that the resulting profile format is the same. For antiSMASH's current state, this format is `HMMER3/f`. Converting the format can be done with `hmmconvert --outfmt 3/f <hmmfile>`, replacing the version with whichever format is required. The format of an HMM file is included in the first line of the file.

Once the profile is created, place the resulting file in the detection module's data directory, the current location for antiSMASH is [antismash/detection/hmm\_detection/data](https://github.com/antismash/antismash/tree/master/antismash/detection/hmm_detection/data). Within that same directory is a file name `hmmdetails.txt`. This file contains a tab-separated line for each HMM profile, in the format `<name> <description> <cutoff> <filename>`. The profile name is the same as given in the `NAME` field within the HMM file and will be the name used within a rule's definition to refer to that HMM profile. The appropriate cutoff is given as an integer and, if not already determined, should ideally minimise false positives and maximise true positives.

### Creating dynamic profiles

Dynamic profiles are defined in python code. These current location of these profiles in antiSMASH, including some existing profiles to use as examples, is [antismash/detection/hmm\_detection/dynamic\_profiles](https://github.com/antismash/antismash/tree/master/antismash/detection/hmm_detection/dynamic_profiles), with one file per profile. The profiles must subclass `antismash.common.hmm_rule_parser.structures.DynamicProfile` and be assigned to a variable `profile` within that file. Once those conditions are satisfied, a dynamic profile is automatically included and will be available for use in a rule's definition.

### Finishing the rule

A rule can be created by adding your new rule text (following the specified grammar mentioned earlier) to one of the rule files. Depending on runtime configuration, not all rule files are used in an antiSMASH run, e.g. the "loose" ruleset is not enabled by default.

Once the rule has been added, running antiSMASH will automatically use that rule if the relevant file is included (e.g. `--hmmdetection-strictness` is set to `loose` to include rules defined in the `loose.txt` within the rules directory.

If any profiles are mentioned within the rules that are not defined in either the `hmmdetails.txt` file or as a dynamic profile, an error will be raised before analysis begins.

If a custom colouring is desired for the new rule type in the HTML output, update [the CSS skeleton](https://github.com/antismash/antismash/blob/master/antismash/outputs/html/css/secmet.scss). To reuse an existing rule's colouring, using "some-new-type" as an example rule name that you wish to have use the same style as "existing-type", add the following to that file:

```
.some-new-type {
    @include existing-type;
}
```

Alternatively, if you want to add completely new colouring, instead add the following to that same file (using the same example name as above):

```
.some-new-type {
    @include secmet(foreground, background, highlight);
}
```

Replace `foreground`, `background`, and `highlight` with any valid CSS colour definitions, e.g. a blue colour could be `blue`, `#0000FF`, or `rgb(0, 0, 255)`.

## Last steps

Test the rule against the known cases and against a few full genomes to ensure that the rule finds the known clusters and has minimal amounts of false positives.

# Detection rule conditions

This page details the elements of a rule's `CONDITIONS` and `EXTENDERS` sections. For details on rule construction as a whole, see [here](creating-a-detection-rule).

The `CONDITIONS` and `EXTENDERS` blocks are formed of one or more individual conditions, each of which requires at least one profile identifier.

Throughout this page, CDS refers to the coding sequence of a gene. In fungi some genes can have splice variants, in which case the translations can vary and the individual coding sequences are important.
