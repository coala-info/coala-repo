You can download the latest WOMtool from the Cromwell releases page on Github . Requirements The following is the toolchain used for development of womtool. Other versions may work, but these are recommended. Scala 2.13 SBT 1.x AdoptOpenJDK 11 HotSpot Git Building sbt assembly will build a runnable JAR in womtool/target/scala-2.13/ Tests are run via sbt test . Note that the tests do require Docker to be running. To test this out while downloading the Ubuntu image that is required for tests, run docker pull ubuntu:latest prior to running sbt test Command Line Usage Run the JAR file with no arguments to get the usage message: $ java -jar womtool.jar java -jar /path/to/womtool.jar &lt;action&gt; &lt;parameters&gt; Actions: validate [--list-dependencies] &lt;WDL file&gt; Performs full validation of the WDL file including syntax and semantic checking. -l or --list-dependencies is an optional flag to list files referenced in import statements. inputs &lt;WDL file&gt; Print a JSON skeleton file of the inputs needed for this workflow. Fill in the values in this JSON document and pass it in to the 'run' subcommand. highlight &lt;WDL file&gt; &lt;html|console&gt; Reformats and colorizes/tags a WDL file. The second parameter is the output type. &quot;html&quot; will output the WDL file with &lt;span&gt; tags around elements. &quot;console&quot; mode will output colorized text to the terminal parse &lt;WDL file&gt; Compares a WDL file against the grammar and writes out an abstract syntax tree if it is valid, and a syntax error otherwise. Note that higher-level AST checks are not done via this sub-command and the 'validate' subcommand should be used for full validation graph &lt;WDL file&gt; Reads a WDL file against the grammar and prints out a .dot of the DAG if it is valid, and a syntax error otherwise. Note that graph currently DOES NOT WORK on version 1.0 workflows. womgraph &lt;WDL file&gt; [ancillary files] Reads a WDL file from the first argument and converts it to a WOM representation then prints out a graph of the WOM produced. Any imported files can be supplied as subsequent arguments. validate Given a WDL file, this runs the full syntax checker over the file and resolves imports in the process. If any syntax errors are found, they are written out. Otherwise the program exits. Error if a call references a task that doesn't exist: $ java -jar womtool.jar validate 2.wdl ERROR: Call references a task (BADps) that doesn't exist (line 22, col 8) call BADps ^ Error if namespace and task have the same name: $ java -jar womtool.jar validate 5.wdl ERROR: Task and namespace have the same name: Task defined here (line 3, col 6): task ps { ^ Import statement defined here (line 1, col 20): import &quot;ps.wdl&quot; as ps ^ --list-dependencies or -l flag For a successful validation, this will output the list of files referenced in import statements in workflows and their subworkflows. $ java -jar womtool.jar validate -l myWdl.wdl Success! List of Workflow dependencies are: /path/to/my/import/myImport.wdl /path/to/another/import/anotherImport.wdl https://path-to-http-import/httpImport.wdl inputs Examine a WDL file with one workflow in it, compute all the inputs needed for that workflow and output a JSON template that the user can fill in with values. The keys in this document should remain unchanged. The values tell you what type the parameter is expecting. For example, if the value were Array[String] , then it's expecting a JSON array of JSON strings, like this: ["string1", "string2", "string3"] $ java -jar womtool.jar inputs 3step.wdl { &quot;three_step.cgrep.pattern&quot;: &quot;String&quot; } This inputs document is used as input to the run subcommand. highlight Formats a WDL file and semantically tags it. This takes a second parameter ( html or console ) which determines what the output format will be. test.wdl task abc { String in command { echo ${in} } output { String out = read_string(stdout()) } } workflow wf { call abc } parse Given a WDL file input, this does grammar level syntax checks and writes out the resulting abstract syntax tree. $ echo "workflow wf {}" | java -jar womtool.jar parse /dev/stdin (Document: imports=[], definitions=[ (Workflow: name=&lt;stdin:1:10 identifier &quot;d2Y=&quot;&gt;, body=[] ) ] ) This WDL file can be formatted in HTML as follows: $ java -jar womtool.jar highlight test.wdl html &lt;span class=&quot;keyword&quot;&gt;task&lt;/span&gt; &lt;span class=&quot;name&quot;&gt;abc&lt;/span&gt; { &lt;span class=&quot;type&quot;&gt;String&lt;/span&gt; &lt;span class=&quot;variable&quot;&gt;in&lt;/span&gt; &lt;span class=&quot;section&quot;&gt;command&lt;/span&gt; { &lt;span class=&quot;command&quot;&gt;echo ${in}&lt;/span&gt; } &lt;span class=&quot;section&quot;&gt;output&lt;/span&gt; { &lt;span class=&quot;type&quot;&gt;String&lt;/span&gt; &lt;span class=&quot;variable&quot;&gt;out&lt;/span&gt; = &lt;span class=&quot;function&quot;&gt;read_string&lt;/span&gt;(&lt;span class=&quot;function&quot;&gt;stdout&lt;/span&gt;()) } } &lt;span class=&quot;keyword&quot;&gt;workflow&lt;/span&gt; &lt;span class=&quot;name&quot;&gt;wf&lt;/span&gt; { &lt;span class=&quot;keyword&quot;&gt;call&lt;/span&gt; &lt;span class=&quot;name&quot;&gt;abc&lt;/span&gt; } graph The syntax of the graph command is: womtool graph [--all] wdlFile.wdl Given a WDL file input, command generates the data-flow graph through the system in .dot format. For example the fork-join WDL: task mkFile { command { for i in `seq 1 1000` do echo $i done } output { File numbers = stdout() } runtime {docker: &quot;ubuntu:latest&quot;} } task grep { String pattern File in_file command { grep '${pattern}' ${in_file} | wc -l } output { Int count = read_int(stdout()) } runtime {docker: &quot;ubuntu:latest&quot;} } task wc { File in_file command { cat ${in_file} | wc -l } output { Int count = read_int(stdout()) } runtime {docker: &quot;ubuntu:latest&quot;} } task join { Int grepCount Int wcCount command { expr ${wcCount} / ${grepCount} } output { Int proportion = read_int(stdout()) } runtime {docker: &quot;ubuntu:latest&quot;} } workflow forkjoin { call mkFile call grep { input: in_file = mkFile.numbers } call wc { input: in_file=mkFile.numbers } call join { input: wcCount = wc.count, grepCount = grep.count } output { join.proportion } } Produces the DAG: digraph forkjoin { &quot;call forkjoin.mkFile&quot; -&gt; &quot;call forkjoin.wc&quot; &quot;call forkjoin.mkFile&quot; -&gt; &quot;call forkjoin.grep&quot; &quot;call forkjoin.wc&quot; -&gt; &quot;call forkjoin.join&quot; &quot;call forkjoin.grep&quot; -&gt; &quot;call forkjoin.join&quot; } The --all flag If this flag is set, all WDL graph nodes become nodes in the generated DAG, even if they are not "executed". Typically this will mean task declarations and call outputs. For example in the above example, with --all you would get: digraph forkjoin { &quot;call forkjoin.grep&quot; -&gt; &quot;String forkjoin.grep.pattern&quot; &quot;call forkjoin.grep&quot; -&gt; &quot;output { forkjoin.grep.count = read_int(stdout()) }&quot; &quot;call forkjoin.grep&quot; -&gt; &quot;File forkjoin.grep.in_file&quot; &quot;call forkjoin.wc&quot; -&gt; &quot;output { forkjoin.wc.count = read_int(stdout()) }&quot; &quot;call forkjoin.grep&quot; -&gt; &quot;call forkjoin.join&quot; &quot;call forkjoin.wc&quot; -&gt; &quot;File forkjoin.wc.in_file&quot; &quot;call forkjoin.mkFile&quot; -&gt; &quot;call forkjoin.grep&quot; &quot;call forkjoin.join&quot; -&gt; &quot;output { forkjoin.join.proportion = read_int(stdout()) }&quot; &quot;call forkjoin.join&quot; -&gt; &quot;Int forkjoin.join.wcCount&quot; &quot;call forkjoin.wc&quot; -&gt; &quot;call forkjoin.join&quot; &quot;call forkjoin.mkFile&quot; -&gt; &quot;output { forkjoin.mkFile.numbers = stdout() }&quot; &quot;call forkjoin.mkFile&quot; -&gt; &quot;call forkjoin.wc&quot; &quot;call forkjoin.join&quot; -&gt; &quot;Int forkjoin.join.grepCount&quot; } Previous Next Copyright © 2026 Broad Institute Built with MkDocs using a theme provided by Read the Docs .
