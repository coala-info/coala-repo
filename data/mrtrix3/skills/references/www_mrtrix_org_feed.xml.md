Here I’ll be focusing exclusively on the story of &lt;code class=&quot;language-plaintext highlighter-rouge&quot;&gt;fixelcfestats&lt;/code&gt;’s excessive RAM usage.&lt;/p&gt; &lt;p&gt;It’s a long story, so zero judgement for anyone who puts this in the TL;DR basket, but I’d prefer for the whole thing to be publicly visible. I’m also hoping that it might provide some insight into how we think about research software engineering problems, even if one can’t fully grasp the relevant code snippets.&lt;/p&gt; &lt;h2 id=&quot;first-awareness-of-issue&quot;&gt;First awareness of issue&lt;/h2&gt; &lt;p&gt;My first encounter with this particular issue occurred upon receiving &lt;em&gt;an invitation to review the manuscript&lt;/em&gt; entitled: “ModelArray: a memory-efficient R package for statistical analysis of fixel data” (&lt;a href=&quot;https://www.biorxiv.org/content/10.1101/2022.07.12.499631v1.full&quot;&gt;Preprint&lt;/a&gt;).&lt;/p&gt; &lt;p&gt;There they reported an inability to perform fixel-based statistical inference using &lt;code class=&quot;language-plaintext highlighter-rouge&quot;&gt;fixelcfestats&lt;/code&gt; due to the RAM usage being highly excessive when using a combination of a very large number of input fixel data files (around 1,000 in their desired use case) and extensive multi-threading.&lt;/p&gt; I further hope that the following content of this post will provide more robust justification for some of the recommendations I made during that process, making up some small fraction for my inability to provide an exhaustive diagnosis at the time.&lt;/p&gt; &lt;h2 id=&quot;basic-operation-of-the-statistical-inference-pipeline&quot;&gt;Basic operation of the statistical inference pipeline&lt;/h2&gt; &lt;p&gt;I’m going to try to give a lay description of how the relevant code works, so that regardless of a reader’s familiarity with software code, the story here has a chance of making sense.&lt;/p&gt; &lt;ol&gt; &lt;li&gt; &lt;p&gt;All input fixel data are explicitly loaded from the various fixel data files that are specified as the input file list, into a single large 2D matrix of numerical data, with the numbers of rows and columns in that matrix corresponding to the numbers of input files and fixels respectively.&lt;/p&gt; &lt;/li&gt; &lt;li&gt; &lt;p&gt;The General Linear Model (GLM) is responsible for deriving, given the input data, design matrix, and contrast of interest, a “test statistic” for each fixel.&lt;/p&gt; &lt;p&gt;This runs independently across multiple threads. Each thread processes an individual &lt;em&gt;shuffling&lt;/em&gt; of the input data. Multiple threads may read from the same input data at the same time, but this does not cause any issues with race conditions.&lt;/p&gt; &lt;/li&gt; &lt;li&gt; &lt;p&gt;The Connectivity-based Fixel Enhancement (CFE) method takes as input the test statistics as computed by the GLM, and “enhances” these values by looking for fixels that both exhibit large test statistics and are highly connected to one another.&lt;/p&gt; &lt;p&gt;This step also runs in a wholly multi-threaded manner. Each thread takes as input a matrix of test statistics produced by the GLM: here, the number of rows is equal to the number of fixels, and each column corresponds to a specific &lt;em&gt;hypothesis&lt;/em&gt; (since multiple hypotheses can be tested with a single command invocation).&lt;/p&gt; &lt;p&gt;The other requisite input data in the case of &lt;code class=&quot;language-plaintext highlighter-rouge&quot;&gt;fixelcfestats&lt;/code&gt; is the fixel-fixel connectivity matrix. From version 3.0.0, this matrix is pre-computed using the &lt;code class=&quot;language-plaintext highlighter-rouge&quot;&gt;fixelconnectivity&lt;/code&gt; command and provided as one of the inputs to &lt;code class=&quot;language-plaintext highlighter-rouge&quot;&gt;fixelcfestats&lt;/code&gt;: https://github.com/MRtrix3/mrtrix3/pull/1543&lt;/p&gt; &lt;/li&gt; &lt;li&gt; &lt;p&gt;Data from across the various threads must be aggregated to form one or more non-parametric null distributions.&lt;/p&gt; &lt;/li&gt; &lt;li&gt; &lt;p&gt;Once generation of the null distributions(s) is completed, data produced from the GLM in step 2, but &lt;em&gt;without&lt;/em&gt; any random shuffling applied, and enhanced using CFE in step 3, are compared to the non-parametric null distributions produced in step 4 to produce a FWE-corrected &lt;em&gt;p&lt;/em&gt;-value for each fixel.&lt;/p&gt; &lt;/li&gt; &lt;/ol&gt; &lt;table&gt; &lt;tbody&gt; &lt;tr&gt; &lt;td&gt;![Screenshot from 2025-08-28 18-06-01&lt;/td&gt; &lt;td&gt;689x388](upload://t2zf08lF57B11PWqXFtBc9zGH0g.png)&lt;/td&gt; &lt;/tr&gt; &lt;/tbody&gt; &lt;/table&gt; &lt;h2 id=&quot;diagnosis-of-the-problem&quot;&gt;Diagnosis of the problem&lt;/h2&gt; &lt;p&gt;Instinctively, there is no intrinsic reason why performing fixel-based statistical inference should become computationally infeasible from the perspective of dynamic memory allocation merely because the number of input fixel data files is large.&lt;/p&gt; &lt;h3 id=&quot;a-priori-expectations&quot;&gt;&lt;em&gt;a priori&lt;/em&gt; expectations&lt;/h3&gt; &lt;p&gt;Regarding the components of the framework as described above:&lt;/p&gt; &lt;ol&gt; &lt;li&gt; &lt;p&gt;The input data are stored as a single 2D matrix and are deliberately &lt;em&gt;not&lt;/em&gt; duplicated across multiple threads. The memory consumed for the storage of the input data should therefore &lt;em&gt;not&lt;/em&gt; scale with the number of executing threads if the implementation is behaving as intended. I was therefore confident that this would not be a source of the problem.&lt;/p&gt; &lt;p&gt;While the total memory consumption of the command could be reduced by storing and accessing the input data in such a way that does not require an explicit load of all input data into RAM (and the ModelArray authors demonstrate how to do so), &lt;em&gt;not&lt;/em&gt; doing so was an explicit design decision in &lt;em&gt;MRtrix3&lt;/em&gt;.
