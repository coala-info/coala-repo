# sra_manifest_to_concatenated_fastqs_parallel

## Overview

This workflow automates the retrieval and organization of sequencing data from the Sequence Read Archive (SRA). It accepts a tabular SRA manifest, such as those generated by the [SRA Run Selector](https://www.ncbi.nlm.nih.gov/Traces/study/), and uses it to batch-download sequencing runs. Users provide the manifest and specify the column indices for SRA run identifiers and the desired final sample names.

The pipeline processes the manifest to map individual runs to their respective experiments. It utilizes `fasterq-dump` to extract reads in parallel and then relabels the datasets based on user-defined identifiers. A key feature of this workflow is its ability to automatically concatenate multiple sequencing runs into a single FASTQ file (or pair) if they belong to the same sample, ensuring a streamlined, per-sample output structure.

The final results are organized into two distinct collections: one for paired-end datasets and another for single-read datasets. This workflow is released under the MIT license and is designed to handle large-scale data retrieval efficiently by splitting the manifest into individual jobs. For detailed instructions on input preparation and parameter settings, refer to the [README.md](./README.md) in the Files section.

## Inputs and data preparation

| Step | Label | Type | Notes |
| --- | --- | --- | --- |
| 0 | SRA_manifest | data_input | Input tabular from SRA Run Selector or home made (First row needs to be a header) |
| 1 | Column number with SRA ID | parameter_input | Column number in the SRA_manifest with SRA ID (usually 1) |
| 2 | Column number with final identifier | parameter_input | Column number in the SRA_list with final identifier |


Ensure the SRA manifest is uploaded as a tabular file with a header line, typically using column 1 for SRA Run IDs and columns like 6, 16, or 36 for sample identifiers. The workflow automatically handles the transition from a single dataset to a collection by splitting the manifest and downloading runs in parallel. For automated testing or execution, you can initialize a job template using `planemo workflow_job_init` to create a `job.yml` file. Refer to the `README.md` for comprehensive details on column selection and character limitations in sample names. One paragraph only.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run sra-manifest-to-concatenated-fastqs.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run sra-manifest-to-concatenated-fastqs.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run sra-manifest-to-concatenated-fastqs.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init sra-manifest-to-concatenated-fastqs.ga -o job.yml`
- Lint: `planemo workflow_lint sra-manifest-to-concatenated-fastqs.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.
