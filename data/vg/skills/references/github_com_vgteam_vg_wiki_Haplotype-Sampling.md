# Haplotype Sampling

[Jump to bottom](#wiki-pages-box)

Adam Novak edited this page Mar 23, 2026
·
[36 revisions](/vgteam/vg/wiki/Haplotype-Sampling/_history)

## Overview

When a pangenome graph is used as a reference for read mapping, the variants that are included in the graph affect the results. Roughly speaking,

* variants that are present **both in the graph and in the sequenced sample** make mapping faster and more accurate; while
* variants that are present in the **graph but not in the sample** make mapping slower and less accurate.

The usual approach is building a graph with only common variants. **Starting with vg version 1.49.0**, there is another option: building a personalized reference for each sample. This is achieved by counting kmers in the reads and generating a small number of synthetic haplotypes based on the kmer counts.

## All-In-One Workflow

As of vg **v1.74.0** (when <https://github.com/vgteam/vg/pull/4849> should be merged), you can do all the k-mer counting, indexing, sampling, and mapping with a single `vg giraffe` command. For example, to replicate the long-read Giraffe workflow from [Rapid, accurate long- and short-read mapping to large pangenome graphs with vg Giraffe](https://doi.org/10.1101/2025.09.29.678807), you can use a command like:

```
vg giraffe --parameter-preset r10 --haplotype-sampling -Z hprc-v2.1-mc-chm13-eval.gbz --no-diploid-sampling --num-haplotypes 16 --set-reference CHM13 -f reads.fq >alignments.gam
```

## Preprocessing the graph

```
vg index -j graph.dist --no-nested-distance graph.gbz
vg gbwt -p --num-threads 16 -r graph.ri -Z graph.gbz
vg haplotypes -v 2 -t 16 -H graph.hapl graph.gbz
```

We assume that the graph is based on a multiple sequence alignment and has a fairly linear high-level structure. In particular, all haplotype paths must participate in the same top-level [chain](https://github.com/vgteam/vg/wiki/Snarls-and-chains). Such graphs can be built using the [Minigraph-Cactus pipeline](https://github.com/ComparativeGenomicsToolkit/cactus/blob/master/doc/pangenome.md) (use the default / `clip` graph). The graph and the haplotypes must be in GBZ format.

The preprocessing step is done once per graph. It transforms each chromosome into a sequence of subchains (blocks) of target length 10 kbp. The idea is that we can choose the most relevant haplotypes independently in each subchain and concatenate them to form synthetic haplotypes. For each subchain, we select some kmers that are specific to the subchain and represent each haplotype by their presence/absence. At the moment, the kmers are minimizers with a single hit in the graph.

**Note:** As of **vg version 1.63.1**, preprocessing will fail if the graph contains components (~chromosomes, contigs) with multiple top-level chains in the snarl decomposition. Such components lack the necessary linear structure.

**Note:** There is an improved preprocessing step starting from **vg version 1.64.0**. It allows sampling fragmented haplotypes in graph regions that have been clipped. Haplotype information (`.hapl`) files built with older versions should be rebuilt to take advantage of the improvements.

### Required indexes

To sample the graph, you only need two files: the GBZ graph and the `.hapl` file with the haplotype information.

To *generate* the haplotype index, you need three indexes in addition to the GBZ graph: a distance index, an r-index, and a minimizer index.

The distance index can be built using the `vg index` [subcommand](/vgteam/vg/wiki/Index-Construction).

The r-index can be built using the `vg gbwt` [subcommand](/vgteam/vg/wiki/VG-GBWT-Subcommand). Option `-p` / `--progress` prints progress information to stderr, while option `--num-threads` specifies the number of threads used in index construction.

We build a new minimizer index instead of loading a prebuilt one from disk. Because we do not need the distance index annotations used by `vg giraffe`, we can save memory by building a smaller index without them.

**Note:** If the graph is complex, distance index construction may require excessive amounts of memory. If you do not need this distance index for any other purpose, you can save memory by building a limited version of the index with option `--no-nested-distance` (or `--snarl-limit 1` instead for vg versions <1.62.0). This will print a warning about oversized snarls, which can be safely ignored. Do **not** use a snarl limit of 0; this will remove all distances, including the top-level chain distances which haplotype sampling needs.

### Generating haplotype information

We generate the haplotype information using the `vg haplotypes` subcommand. There are two required arguments: the input graph that will be preprocessed, and the output file for the haplotype information, specified with option `-H` / `--haplotype-output`.

The following options can be used:

* `-d` / `--distance-index`: Distance index file. Default: `graph.dist` if the input graph is `graph.gbz`.
* `-r` / `--r-index`: R-index file. Default: `graph.ri` if the input graph is `graph.gbz`.
* `--kmer-length`: Kmer length used to be used in the haplotype information (up to 31). Default: 29.
* `--window-length`: Window length used for building the minimizer index. Default: 11.
* `--subchain-length`: Target length (in bp) for subchains. Default: 10000.
* `-v` / `--verbosity`: Verbosity level (0 = silent, 1 = basic, 2 = detailed). Default: 0.
* `-t` / `--threads`: Approximate number of threads to be used. Default: depends on the system.

## Haplotype sampling

Once you have a GBZ graph and a `.hapl` file of haplotype information, you can make a`.kff` file of kmer counts from a sequencing sample, and haplotype-sample the graph to create a personalized pangenome reference for your sample.

You can do this manually, saving the sampled graph to disk:

```
export TMPDIR=/scratch/tmp
kmc -k29 -m128 -okff -t16 -hp sample.fq.gz ${TMPDIR}/sample $TMPDIR
vg haplotypes -v 2 -t 16 --include-reference --diploid-sampling \
    -i graph.hapl -k ${TMPDIR}/sample.kff -g ${TMPDIR}/sampled.gbz graph.gbz
vg giraffe -p -t 16 -Z ${TMPDIR}/sampled.gbz -i -f sample.fq.gz > sample.gam
```

You can also use the haplotype sampling integration in Giraffe to do make the sampled graph during the read alignment process (requires **vg version 1.51.0**):

```
export TMPDIR=/scratch/tmp
kmc -k29 -m128 -okff -t16 -hp sample.fq.gz ${TMPDIR}/sample $TMPDIR
vg giraffe -p -t 16 -Z graph.gbz --haplotype-name graph.hapl --kmer-name ${TMPDIR}/sample.kff \
    -N sample -i -f sample.fq.gz > sample.gam
```

Haplotype sampling is based on kmer counts in the reads. It assumes a diploid sample. Using the counts, the algorithm classifies each kmer used in the haplotype information as absent, heterozygous, present, or frequent in the sequenced sample. It then selects a small number of the most relevant haplotypes in each subchain and concatenates them to form synthetic haplotypes.

**Note:** Read coverage should be at least 20x to tell the difference between heterozygous and homozygous kmers reliably.

### Kmer counting

We assume that the kmer counts are in the [KFF format](https://github.com/Kmer-File-Format/kff-reference). Any kmer counter supporting the format should work, though we have used only [KMC](https://github.com/refresh-bio/KMC) so far. Kmers with a single occurrence in the reads can be safely ignored, and the counts for a kmer and its reverse complement can be combined or separate.

It is important to use the same kmer length in kmer counting as in the haplotype information (default: 29).

### Sampling the haplotypes

We use the `vg haplotypes` for sampling the haplotypes. In addition to the input graph, we need the haplotype information and the kmer counts we generated earlier. These can be specified using options `-i` / `--haplotype-input` and `-k` / `--kmer-input`, respectively. The output is a sampled GBZ graph, which will be written to a file specified by option `-g` / `--gbz-output`.
