# COVID-19: consensus construction

## Overview

This workflow generates high-quality SARS-CoV-2 consensus sequences from variant calls and aligned reads. It is designed to work seamlessly with outputs from the [IWC variant calling workflows](https://github.com/galaxyproject/iwc/tree/main/workflows/sars-cov-2-variant-calling) and is part of the [covid19.galaxyproject.org](https://covid19.galaxyproject.org) effort. The process ensures that consensus sequences are built using transparent, configurable criteria for variant inclusion and masking.

The workflow processes VCF files and BAM alignments to apply specific filters. It incorporates "FILTER PASS" variants that meet a user-defined allele frequency (AF) threshold into the consensus. Conversely, it hard-masks (using N) genomic positions where coverage falls below a specified depth or where variants exist with an AF between the consensus and failure thresholds. This approach prevents the inclusion of ambiguous or poorly supported sites while preserving high-confidence variants even in low-coverage regions.

Key tools utilized include `bcftools consensus`, `bedtools Genome Coverage`, and `SnpSift`. The final outputs include individual consensus sequences for each sample and a combined multisample FASTA file. For detailed information on input preparation and parameter settings, please refer to the [README.md](README.md) in the Files section.

## Inputs and data preparation

| Step | Label | Type | Notes |
| --- | --- | --- | --- |
| 0 | Variant calls | data_collection_input | Collection of VCFs produced by upstream workflows for variation analysis |
| 1 | min-AF for consensus variant | parameter_input | Only variant calls with an allele-frequency greater this value will be considered consensus variants. |
| 2 | min-AF for failed variants | parameter_input | Variant calls with an allele frequency higher than this value, but lower than the AF threshold for consensus variants will be considered questionable and the respective sites be masked (with Ns) in the consensus sequence. |
| 3 | aligned reads data for depth calculation | data_collection_input | Fully processed BAMs as generated by upstream workflows for variation analysis. Note: for ARTIC data, these BAMs should NOT have undergone processing with ivar removereads. |
| 4 | Depth-threshold for masking | parameter_input | Sites in the viral genome covered by less than this number of reads are considered questionable and will be masked (with Ns) in the consensus sequence independent of whether a variant has been called at them or not. |
| 5 | Reference genome | data_input | The SARS-CoV-2 reference genome |


To prepare your data, ensure variant calls are provided as a VCF collection containing DP and DP4 INFO fields, paired with a collection of aligned reads in BAM format and a FASTA reference genome. Use data collections rather than individual datasets to maintain sample-to-sample mapping between VCFs and BAMs during the consensus building process. You can adjust the allele frequency and depth thresholds to control hard-masking (N) for low-coverage or ambiguous sites. For automated job configuration, consider using `planemo workflow_job_init` to generate a `job.yml` template. Refer to the README.md for comprehensive details on how specific filter criteria and masking logic are applied to the final multisample FASTA.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run consensus-from-variation.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run consensus-from-variation.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run consensus-from-variation.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init consensus-from-variation.ga -o job.yml`
- Lint: `planemo workflow_lint consensus-from-variation.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
