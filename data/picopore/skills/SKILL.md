# picopore

## Overview
Picopore is a compression utility specifically designed for Oxford Nanopore FAST5 files. It helps researchers manage the massive data volumes generated by nanopore sequencing by removing redundant intermediate datasets or applying HDF5-level compression. While modern ONT basecallers now implement some of these strategies natively, Picopore is highly effective for "raw" compression—which strips "squiggle-space" data to save significant space—and for deep-lossless archival of 2D datasets.

## Common CLI Patterns

### Compression Modes
Picopore requires the selection of one of three modes:

*   **Lossless**: Uses HDF5's internal gzip functionality. Best for general footprint reduction while maintaining full tool compatibility.
    ```bash
    picopore --mode lossless /path/to/fast5/
    ```
*   **Deep-Lossless**: Indexes basecalled datasets to event detection. Offers higher compression for 2D datasets but requires a revert step before using most downstream tools.
    ```bash
    picopore --mode deep-lossless /path/to/fast5/
    ```
*   **Raw**: The most aggressive mode. It removes event detection and basecall data, leaving only the raw signal and configuration.
    ```bash
    picopore --mode raw --fastq --summary /path/to/fast5/
    ```

### Reverting Compressed Files
If you used `lossless` or `deep-lossless` modes and need to restore the files to their original state for compatibility with specific tools:
```bash
picopore --mode deep-lossless --revert /path/to/fast5/
```

### Real-time Compression
To compress reads as they are generated during a sequencing run, use the `picopore-realtime` command to monitor an output directory:
```bash
picopore-realtime --mode raw --threads 4 /path/to/output/reads/
```

### Manual Group Removal
For advanced users, you can manually specify groups to remove using regular expressions in `raw` mode:
```bash
picopore --mode raw --manual "1D.*Events" /path/to/fast5/
```

## Expert Tips and Best Practices

*   **Check Native Compression First**: As of May 2017, ONT basecallers (like Albacore/Guppy/MinKNOW) often implement native HDF5 compression. If your files are recent, `lossless` mode may provide minimal gains. Use `raw` mode if you only need the raw signal for future re-basecalling.
*   **Verify Integrity**: Always use `picopore-test` after a compression run to ensure that datasets and attributes remain equal (applicable to lossless modes).
*   **Tool Compatibility**: 
    *   `lossless` files work with almost all tools (Metrichor, nanopolish, poretools).
    *   `raw` files are incompatible with tools that require "squiggle-space" data, such as `nanopolish` or `nanoraw`.
*   **System Dependencies**: Ensure `h5repack` (from `hdf5-tools`) is installed on your system path, as Picopore relies on it for the underlying HDF5 manipulations.
*   **Parallel Processing**: Use the `-t` or `--threads` flag to speed up processing on multi-core systems, especially when dealing with thousands of individual FAST5 files.
