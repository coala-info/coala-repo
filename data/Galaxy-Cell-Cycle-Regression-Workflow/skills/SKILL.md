# Cell Cycle Regression Workflow

## Overview

This Galaxy workflow is designed to mitigate the biological noise introduced by cell cycle variation in single-cell RNA-sequencing (scRNA-seq) data. Using the [Scanpy](https://scanpy.readthedocs.io/) ecosystem, it processes an AnnData object that has already undergone quality control, normalization, and scaling. The pipeline identifies cell cycle phases by scoring cells based on provided S-phase and G2M-phase gene lists, allowing for the systematic removal of these effects to reveal underlying developmental or experimental signals.

The workflow utilizes `Scanpy RegressOut` to mitigate the influence of cell cycle scores and other unwanted variables. It includes several data manipulation steps using `anndata_inspect` and `table_compute` to manage metadata and gene lists. Following regression, the pipeline performs dimensionality reduction and clustering via Louvain or Leiden algorithms to compare the data structure before and after the correction.

Final outputs include regressed AnnData objects and visualization plots (UMAP/tSNE) generated by `scanpy_plot`. These visualizations help researchers verify that the clustering is driven by biological cell types rather than the proliferative state of the cells. Ensure your primary input is a high-quality AnnData (h5ad) file that has already undergone quality control, normalization, and scaling to ensure accurate regression. The gene lists for S and G2M phases should be provided as tabular files or pasted text, matching the gene identifiers used in your AnnData object. While this workflow processes individual datasets, ensure all input metadata is consistent to avoid errors during the Scanpy manipulation steps. For automated execution and parameter testing, you can use `planemo workflow_job_init` to generate a `job.yml` file. Please refer to the README.md for comprehensive details on specific tool parameters and data preparation requirements.

## How to run (Planemo)

### Local Galaxy (default engine)

```bash
planemo run main-workflow.ga job.yml --download_outputs --output_directory . --output_json output.json
```

Optional: `--galaxy_root /path/to/galaxy` speeds up spinning up a local Galaxy.

### Docker Galaxy

Append e.g. `--engine docker_galaxy --ignore_dependency_problems` (requires Docker).

### External Galaxy (usegalaxy.org / EU / …)

```bash
planemo run main-workflow.ga job.yml --engine external_galaxy --galaxy_url SERVER_URL --galaxy_user_key YOUR_API_KEY
```

Create a reusable profile: `planemo profile_create myprofile --galaxy_url ... --galaxy_user_key ... --engine external_galaxy` then `planemo run main-workflow.ga job.yml --profile myprofile`.

### Job file

- Generate a template: `planemo workflow_job_init main-workflow.ga -o job.yml`
- Lint: `planemo workflow_lint main-workflow.ga`

See **[Running Galaxy workflows](https://planemo.readthedocs.io/en/master/running.html)** in the Planemo documentation.

## Invocation outputs

After a successful run with `--download_outputs`, files appear under your chosen `--output_directory`. `--output_json` records CWL-style metadata for outputs (paths, checksums).

On an external Galaxy, use `planemo invocation_download INVOCATION_ID --profile ...` to fetch outputs, or track progress with `planemo workflow_track INVOCATION_ID`.

Workflow steps marked as **workflow outputs** in the editor are listed under **Workflow outputs**; exact filenames depend on tools and Galaxy configuration.
